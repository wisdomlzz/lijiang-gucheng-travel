# 首页功能分类方案 — 分析与实施计划

> 来源：`docs/requirements/active/小程序需求与资质梳理.md` 需求点02

## 背景

小程序面向两类用户（游客/商户），需在同一入口下通过分类宫格让不同角色快速定位功能。客户不倾向拆分为两个小程序。

## 成熟案例分析（市面参考）

### 四、分类规律提炼

从以上案例中可以归纳出几条行业共性：

1. **按场景而非按角色分**：绝大多数景区小程序在首页宫格层面不按"游客/商户"做一级分类，而是按"游玩场景"分（票务、导览、消费、服务）。商户功能通常通过"角色权限"在同一入口下隐藏/展示，而非在首页显式分区。

2. **分类数量集中在 4–6 个**：宫格一级分类通常 4–6 个，太多则用户认知负担重，太少则功能覆盖不足。其中"我的/个人"几乎必占一个。

3. **"导览/游览"类永远是第一入口**：几乎所有案例的第一分类都是与"游览"直接相关的功能（购票、导览、地图），体现"游客到了先想怎么玩"的核心诉求。

4. **"便民/服务"类是通用配置**：乌镇、西湖、敦煌均有"服务"分类，涵盖客服、投诉、公告、天气、交通等。凤凰古城将"便民服务"扩展到商户管理层面（一键服务）。

5. **本地特色需要独立入口**：有独特功能的景区通常将特色单独成类——敦煌的"数字体验"、黄山的"VR 全景"、丽江的"文化院落打卡"和"AI 萌蛙咨询"都是这种逻辑。

6. **商户功能隐于权限，不占首页宫格**：WeTravel、凤凰古城的商户功能都通过后台/管理端实现，不在游客首页宫格中展示。这与客户"一个入口、按权限呈现"的诉求一致。

### 当前方案的覆盖度评估

**覆盖较好的部分**："游玩"对应导览/票务/打卡等游览功能；"古城"承载本地特色（文化院落、文创）；"办事"涵盖支付/退款/发票等便民服务；"我的"处理个人中心和订单管理。四分类基本能映射需求点 01/03/04/06。

**潜在缺口与建议**：

- "游玩"与"古城"的边界模糊：案例规律第 1 条指出应按"场景"分而非按"主题"分。"游玩"和"古城"在用户认知上可能重叠——导览地图、票务是"游玩"还是"古城"？建议明确划分标准，比如"游玩"= 通用旅游服务（票务、导览、地图、攻略），"古城"= 丽江特色内容（文化院落打卡、文创、古城故事）。

- 缺少"资讯/公告"的独立出口：需求文档中多次提到公告、预警、运营资讯。多数案例（乌镇、西湖、敦煌）都有独立的"资讯/信息"入口。若不加独立分类，需确认资讯内容放在哪里（Banner 轮播？某个分类的二级页？）。

- AI 萌蛙咨询的入口方式待定：案例中黄山将智能客服放在首页浮动按钮，智游丽江将其收入"智能推荐"分类。建议萌蛙作为首页常驻浮动入口，不依赖分类宫格，保证任何页面都能唤起。

- 商户功能的可见性设计：需求点 05（运营管理系统）和部分需求点 04/06 的商户侧功能，按案例规律应走权限控制而非独立分类。需在 Phase 2 明确"商户登录后首页多看到什么、少看到什么"。

## 推荐方案：游玩 · 古城 · 服务 · 我的

### 命名调整（对比当前方案）

| 当前方案 | 推荐方案 | 调整理由 |
|---------|---------|---------|
| 游玩 | **游玩** | 保持不变。对应案例规律第 3 条"导览/游览永远是第一入口" |
| 古城 | **古城** | 保持不变。承载丽江独有的文化身份，对应规律第 5 条"本地特色独立入口" |
| 办事 | **服务** | **改名**。"办事"偏行政化，游客看到"办事"会产生距离感。"服务"是行业通用词（乌镇、西湖、敦煌均用"服务"），更亲切，且自然涵盖便民服务、投诉、公告等所有场景 |
| 我的 | **我的** | 保持不变。行业标准 |

### 核心边界定义

**"游玩" vs "古城"怎么分？**

- **游玩 = 游客的通用行为**：到了一个地方，要导航、找吃的、找住的、买票、停车、逛商城。这些功能放在任何景区都能用。
- **古城 = 丽江独有的内容**：文化院落打卡、遗产知识探索、古城故事与文化期刊、古城维护费说明。离开丽江就没有这些功能。

一句话记忆：**"怎么玩"归游玩，"丽江是什么"归古城。**

### 每个分类下的功能清单（18 个现有宫格项）

#### 游玩（Discover）— 游客的通用游览工具（6 项）

| # | 宫格项 | 说明 |
|---|-------|------|
| 1 | 导览地图 | Leaflet 地图，POI 标注，分类侧边栏，收藏 |
| 2 | 讲解服务 | 导游讲解服务预订/语音导览 |
| 3 | 精选路线 | 精选/推荐路线，逐站导航 |
| 4 | 停车服务 | 停车场列表、余位信息、导航 |
| 5 | 门票预定 | 门票购买（跳转有赞） |
| 6 | 联票套餐 | 联票/套餐/千古情等组合产品 |

划分标准：这些功能放在任何景区都能用——导航、买票、找路线、停车，是"怎么玩"的通用工具。

#### 古城（Heritage）— 丽江独有的文化体验（5 项）

| # | 宫格项 | 说明 |
|---|-------|------|
| 1 | 文化院落 | 院落列表 + 详情，打卡、VR 全景、语音导览、已打卡标识 |
| 2 | 遗产知识 | 8 类 Tab（道路/水系/古井/古桥/古树/民居/古建/文化环境） |
| 3 | VR游览 | 古城 VR 全景虚拟游览 |
| 4 | 古城资讯 | 古城文化/东巴文化/民风民俗/纳西古乐等资讯 |
| 5 | 购在古城 | 文创产品、纳西特色手工艺品等文化消费 |

划分标准：离开丽江就没有的内容——世界遗产、文化院落、纳西文化、在地文创消费，是"丽江是什么"的文化身份。

> **关于"官方商城"与"购在古城"的区分**：官方商城归入"游玩"（通用电商，游客买纪念品/周边），购在古城归入"古城"（侧重文创/非遗/在地文化消费）。如果实际业务中两者区分不大，可合并，需与客户确认。

### 首页整体布局

```
┌─────────────────────────────────┐
│  🔍 搜索  景点/商家/路线        │  ← 全局搜索栏
├─────────────────────────────────┤
│                                 │
│     [ Banner 轮播区域 ]          │  ← 运营可配置 Banner
│                                 │
├─────────────────────────────────┤
│ ┌─────────┐  ┌─────────┐       │
│ │  🏔️     │  │  🏛️     │       │
│ │  游玩   │  │  古城   │       │  ← 四宫格 2×2
│ └─────────┘  └─────────┘       │
│ ┌─────────┐  ┌─────────┐       │
│ │  🛎️     │  │  👤     │       │
│ │  服务   │  │  我的   │       │
│ └─────────┘  └─────────┘       │
├─────────────────────────────────┤
│  古城资讯与便民信息               │
│  ┌──────┐ ┌──────┐ ┌──────┐    │  ← 资讯信息流
│  │ 资讯1 │ │ 资讯2 │ │ 资讯3 │    │     (卡片横滑)
│  └──────┘ └──────┘ └──────┘    │
├─────────────────────────────────┤
│                                 │
│                        🐸       │  ← 萌蛙浮动按钮
│                                 │       (常驻右下角)
└─────────────────────────────────┘
```

布局说明：

- **搜索栏**（顶部）：解决"用户已有明确目标"的场景，避免在宫格中翻找。
- **Banner**（运营位）：承接公告、活动、推荐路线等运营内容的展示。与公告列表互补——Banner 是精选推荐，公告列表是全量信息。
- **四宫格**（核心区）：4 个分类入口，点击进入各分类的二级宫格页（每个分类下 6–7 个功能入口）。
- **资讯信息流**（底部）：已实现的资讯聚合区，作为宫格之外的补充浏览区域。
- **萌蛙浮动按钮**（右下角）：对应案例规律第 5 条 + 智游黄山的做法，AI 咨询不占宫格位，作为常驻浮动入口，任何页面可唤起。

### 商户权限设计

按案例规律第 6 条（商户功能隐于权限），商户不单独占宫格。登录后差异化展示如下：

| 位置 | 游客看到 | 商户额外看到 |
|------|---------|-------------|
| 首页宫格 | 4 个分类 | 同上（不变） |
| 服务 > 一键便民 | 下单/查订单 | 额外显示"我的接单"入口（服务人员视角） |
| 我的 | 个人信息/订单/收藏 | 额外显示：诚信评分、收入统计、证件管理、商家发布 |
| 导览地图 | 所有 POI | 同上（不变） |
| 公告通知 | 阅读公告 | 额外显示"商户通知"分类 |

原则：商户登录后首页"看起来一样"，只是在对应分类内多出与经营相关的功能入口。游客完全感知不到这些功能的存在。

### 方案优势总结

1. **命名调整最小、认知成本最低**：仅将"办事"改为"服务"，与行业惯例对齐，降低用户理解成本。
2. **边界清晰、可执行**："怎么玩"vs"丽江是什么"的划分标准简单直接，后续新增功能也能快速归类。
3. **所有现有功能 100% 覆盖**：50+ 页面全部有归属，无遗漏、无重叠。
4. **商户不感知**：同一入口、同一首页、同一宫格，商户通过权限在分类内部获得额外入口，符合客户"不拆两个小程序"的诉求。
5. **萌蛙独立浮动**：解决 AI 咨询"放在哪个分类"的争议，参考黄山/智游丽江的做法，任何页面一键唤起。
