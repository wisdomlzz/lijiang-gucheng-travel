# 丽江古城游平台 · 游客与商户功能承载方案分析

---

## 一、概述

丽江古城游平台同时面向两类核心用户群体：**游客**（前来古城游览）与**商户**（在古城内从事经营活动）。两类用户的功能需求存在差异，部分功能呈现方式亦不相同。

关于商户功能应以何种方式承载，现提出两套方案供审议：

- **方案一**：统一平台，按用户角色动态呈现功能
- **方案二**：单端双模式，登录选端 + 随时切换

本文档客观陈述两套方案的设计逻辑、业务流程及各自优劣，供决策参考。

---

## 二、用户群体与功能需求分析

### 2.1 游客功能需求

游客的核心诉求为游览体验与旅途服务，主要功能如下：

| 需求类别 | 具体功能 |
|---------|---------|
| 游览导览 | 导览地图、文化院落、遗产知识、VR 全景游览、精选路线 |
| 古城消费 | 购在古城（商家浏览）、古城资讯、便民信息 |
| 旅途服务 | 一键服务（行李搬运、送货）、订单管理 |
| 互动反馈 | AI 咨询、一键投诉、随手拍 |
| 权益激励 | 文化院落打卡、积分中心 |
| 信息获取 | 公告通知、消息 |

### 2.2 商户功能需求

商户的核心诉求为店铺经营与日常便民服务，主要功能如下：

| 需求类别 | 具体功能 |
|---------|---------|
| 店铺管理 | 我的店铺（信息维护、营业状态、变更审核） |
| 经营服务 | 一键服务（生活垃圾清运、建筑垃圾清运、布草配送、送水服务）、订单管理 |
| 资产管理 | 公房服务（租赁状态、租金缴纳） |
| 活动参与 | 纳西人打卡、积分中心 |
| 信息获取 | 公告通知、消息 |
| 反馈渠道 | 投诉 |

### 2.3 功能重叠分析

经梳理，两类用户的功能重叠情况如下：

- **功能名称相同、内容不同**：一键服务。游客使用行李搬运、送货等点对点服务；商户使用垃圾清运、布草配送、送水等片区型服务。
- **功能名称相同、场景不同**：投诉。游客投诉商家；商户投诉平台。两端各设投诉入口，功能一致。
- **功能名称与内容均相同**：公告通知、消息。两端各自呈现，共享同一数据源。
- **互不涉及的功能**：游客的导览地图、VR 游览、精选路线等纯游览功能，与商户的店铺管理、公房服务等纯经营功能，彼此无关联。

综上，两类用户功能**不存在需要以"通用"名义模糊处理的部分**，应严格按照角色进行功能分配，两端各自完整。

## 三、方案一：统一平台 · 角色化呈现

### 3.1 方案概述

游客与商户使用**同一个小程序**。平台依据登录用户的角色，动态呈现相应功能区。首页功能划分为「游客服务」「商户服务」「通用」三个分区，系统按角色控制各分区的显示。

### 3.2 账号与权限体系

采用单一账号体系，以角色字段区分用户类型：

| 用户类型 | 角色 | 可见功能区 |
|---------|------|-----------|
| 游客 | 游客 | 游客服务、通用 |
| 商户 | 商户 | 商户服务、通用（游客服务亦可见） |

商户角色保留游客服务的访问权限，因其仍可使用游览类功能（如浏览资讯、参与院落打卡）。

### 3.3 商家入驻与权限开通流程

```
游客于首页点击「商家入驻」
        │
        ▼
   提交店铺资料申请（申请状态：待审核）
        │
        ▼
   平台管理员于后台审核
        │
        ├── 驳回 ──→ 申请结束，用户维持游客身份
        │
        ▼ 通过
   用户角色升级为商户
   首页自动解锁「商户服务」分区
   （无需切换应用，原地生效）
```

该流程的特点是**权限升级在应用内原地完成**，用户无需下载新应用或切换入口，体验连贯。

### 3.4 功能呈现方式

- 功能以分区形式集中呈现，商户登录后首页同时显示游客服务、商户服务、通用三个分区。
- 对于「一键服务」等名称相同但内容不同的功能，采取**同一入口、按角色过滤**的方式：系统识别用户角色后，呈现对应的服务类型。

### 3.5 数据管理

采用单一数据域，所有功能数据（订单、积分、投诉、公告等）统一存储与管理，无需跨端同步机制。

### 3.6 方案优势

1. **单一应用**，用户无需安装与切换多个小程序，使用门槛低。
2. **入驻流程连贯**，游客升级为商户后原地解锁功能，无中断。
3. **数据天然统一**，无需建立跨端同步机制，维护成本低。
4. **开发与运维投入较小**。

### 3.7 方案局限

1. **商户首页信息密度较高**，游客服务、商户服务、通用三区并存，商户定位自身所需功能需一定浏览成本。
2. **商户经营场景的聚焦度不足**，游览类功能与经营类功能混合呈现，缺乏专属工作台的体验。
3. **双视角功能（如一键服务）的交互相对复杂**，需在同一入口依据角色区分内容。
4. **"通用"分区概念模糊**，投诉、积分、公告等功能归属不清，两端场景差异被掩盖。

## 四、方案二：单端双模式 · 登录选端 + 随时切换

### 4.1 方案概述

游客与商户使用**同一个小程序**，但在登录前选择角色模式（游客/商户），登录后进入对应首页和底部导航体系。两种模式各自独立、界面分明，商户可随时切回游客端。

核心思路：**对用户而言是两个端，对开发而言是一个应用。**

### 4.2 交互流程

```
打开小程序
        │
        ▼
   登录页（顶部有「游客 / 商户」模式切换）
        │
        ├── 选择「游客」→ 登录 → 游客首页 + 游客导航
        │                         底部：首页 / 一键服务 / AI咨询 / 商城 / 我的
        │
        └── 选择「商户」→ 登录 → 商户首页 + 商户导航
                                 底部：工作台 / 一键服务 / 我的
```

**商户切回游客端**：通过「我的」页面入口或全局切换控件，一键切回游客端界面，无需重新登录。

### 4.3 账号与权限体系

采用单一账号体系，以角色字段区分用户类型：

| 用户类型 | 默认模式 | 可切换 |
|---------|---------|-------|
| 游客 | 游客模式 | 不可切换 |
| 商户 | 商户模式 | 可切换至游客模式 |

商户默认进入商户模式（经营场景为主），但保留游客模式的访问权限，可随时切回使用游览功能。

### 4.4 商家入驻与权限开通流程

```
用户于游客模式下点击「商家入驻」
        │
        ▼
   提交店铺资料申请（申请状态：待审核）
        │
        ▼
   平台管理员于后台审核
        │
        ├── 驳回 ──→ 申请结束，用户维持游客身份
        │
        ▼ 通过
   用户角色升级为商户
   下次登录默认进入商户模式
   登录页可切换至商户模式
   （无需安装新应用，单应用内切换）
```

该流程的特点是**权限升级后用户获得独立商户模式入口**，商户经营场景与游览场景在应用内物理分离，但无需安装额外小程序。

### 4.5 功能呈现方式

两端功能**严格按角色分配**，无"通用"分区，各自完整独立：

**游客模式功能集：**

| 需求类别 | 具体功能 |
|---------|---------|
| 游览导览 | 导览地图、文化院落、遗产知识、VR 全景游览、精选路线 |
| 古城消费 | 购在古城、古城资讯、便民信息 |
| 旅途服务 | 一键服务（行李搬运、送货）、订单管理 |
| 互动反馈 | AI 咨询、投诉、随手拍 |
| 权益激励 | 院落打卡、积分中心 |
| 信息获取 | 公告通知、消息 |

底部导航：首页 / 一键服务 / AI咨询 / 商城 / 我的

**商户模式功能集：**

| 需求类别 | 具体功能 |
|---------|---------|
| 店铺管理 | 我的店铺（信息维护、营业状态、变更审核） |
| 经营服务 | 一键服务（生活垃圾清运、建筑垃圾清运、布草配送、送水服务）、订单管理 |
| 资产管理 | 公房服务（租赁状态、租金缴纳） |
| 活动参与 | 纳西人打卡、积分中心 |
| 信息获取 | 公告通知、消息 |
| 反馈渠道 | 投诉 |

底部导航：首页 / 一键服务 / 我的

**关键设计原则：**

- **不设"通用"分区**：投诉、积分中心、公告通知、消息等功能虽两端均有，但严格归属各自模式，不再以"通用"名义模糊处理。
- **模式即隔离**：一键服务等功能在两种模式下各自独立，无需在同一入口做角色过滤，模式切换即为内容切换。
- **组件复用、数据共享**：投诉、公告、消息等功能的界面组件在两种模式间复用，数据源统一，避免重复开发与数据同步问题。

### 4.6 数据管理

采用单一数据域，所有功能数据统一存储与管理。公告通知、消息等两端共见的数据天然一致，无需跨端同步机制。积分中心虽两端各有入口，但积分获取途径不同（游客通过院落打卡，商户通过纳西人打卡），数据在统一体系内按来源区分。

### 4.7 方案优势

1. **商户模式功能聚焦**，打开即为工作台 + 经营管理功能，无游览类功能干扰，场景清晰。
2. **双视角功能处理简洁**，一键服务等功能在两种模式各自独立，无需角色过滤。
3. **单一应用**，用户无需安装多个小程序，使用门槛低。
4. **入驻流程连贯**，角色升级后在同一应用内切换模式，无中断。
5. **数据天然统一**，单一数据域，无需跨端同步机制。
6. **组件复用降低开发成本**，通用功能的界面组件在两种模式间复用，避免双端重复开发。
7. **两端可独立演进**，商户模式未来扩展经营类功能（如结算管理、财务报表）不影响游客模式。
8. **符合行业通行做法**，商户获得专属工作台体验，与主流本地生活平台的商家端体验一致。

### 4.8 方案局限

1. **模式切换存在认知成本**，商户需理解"游客模式"与"商户模式"的概念，但影响可控（默认进入商户模式，游览时再切换）。
2. **商户模式下无法直接访问游览功能**，需切换模式后方可使用，但实际场景中商户以经营为主、游览为辅，切换频率低。
3. **两种模式需各自维护首页与导航**，较统一平台略增开发量，但远低于双小程序方案。

## 五、方案对比

| 对比维度 | 方案一：统一平台 | 方案二：单端双模式 |
|---------|:--------------:|:--------------:|
| 应用数量 | 一个小程序 | 一个小程序 |
| 商户功能聚焦度 | 低（三分区混合呈现） | 高（独立工作台 + 专属导航） |
| 双视角功能处理 | 同入口按角色过滤 | 各模式独立设置 |
| 通用功能处理 | 设"通用"分区（归属模糊） | 严格按角色分配，无通用分区 |
| 商家入驻体验 | 应用内原地升级 | 应用内切换模式 |
| 通用功能开发成本 | 一套 | 一套（组件复用） |
| 数据一致性保障 | 天然统一（单一数据域） | 天然统一（单一数据域） |
| 商户使用门槛 | 低 | 低（默认进商户模式） |
| 商户切换游览功能 | 直接可见 | 需切换模式 |
| 未来扩展灵活性 | 中 | 高（两种模式独立演进） |
| 行业惯例契合度 | 一般 | 较高 |

## 六、决策考量要素

两套方案各有侧重，适合不同的产品定位与发展预期。决策时建议重点考量以下要素：

**1. 商户的角色定位**

若商户被视为"兼具经营属性的古城参与者"，其游览类功能需要高频使用，则统一平台的混合呈现可满足需求；若商户被视为"以经营为核心的专业用户"，游览类功能对其属于低频需求，则单端双模式的聚焦体验更为契合。

**2. 商户功能的扩展规划**

若商户功能预计保持现有规模（店铺管理、公房、便民服务、纳西人打卡等），统一平台足以承载；若预计将持续扩展经营类功能（如结算管理、财务报表、员工管理、库存管理等），单端双模式可提供独立的扩展空间而不影响游客端体验。

**3. 信息架构的清晰度要求**

若对首页信息架构的清晰度要求不高，可接受三区分区混合呈现，则统一平台更简单；若希望每种角色都有清晰、聚焦的功能入口与导航体验，则单端双模式更优。

**4. 投入产出预期**

两种方案的开发投入差距不大（单端双模式需维护两套导航和首页，但省去了角色过滤逻辑和通用分区的复杂度），上线周期相当。需结合产品定位与用户体验优先级综合权衡。
