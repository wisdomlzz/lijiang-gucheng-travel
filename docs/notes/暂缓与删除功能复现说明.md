# 暂缓与删除功能复现说明

> 口径说明：本文只沉淀历史功能的可取设计，方便后续独立立项时复现。志愿服务当前只保留占位入口；未澄明的业务运行代码、后台 CRUD 和本地 mock 数据不纳入本期 demo。

## 1. 志愿服务（暂缓）

当前状态：C 端首页保留宫格占位入口，进入 `/c/volunteer` 后提示“需求待澄明”；资讯聚合、Desktop 菜单、报名审核和时长记录等业务功能暂不展开。

可复用设计：

| 模块 | 可复用点 |
| --- | --- |
| C 端活动列表 | 活动状态筛选、报名人数/名额展示、活动地点与联系人展示 |
| 活动详情 | 活动说明、封面图、报名按钮、活动状态提示 |
| Desktop 管理 | 活动新增、编辑、上下线、取消、列表检索 |

后续复现建议：

1. 先明确志愿服务的主管部门、招募对象、报名审核规则和服务时长记录规则。
2. 数据模型建议包含：活动标题、活动类型、服务地点、开始/结束时间、招募人数、报名人数、联系人、状态、报名审核状态、签到/签退记录。
3. C 端只展示已发布活动；报名后应有“待审核/已通过/未通过/已取消”的个人报名记录。
4. Desktop 需要具备活动发布、报名审核、签到导出、服务记录统计。
5. 未明确管理责任前，只保留占位入口，不进入招募、报名、审核、时长记录等业务链路。

## 2. C 端商城订单展示（移出本期）

当前状态：已删除本地商品订单 store、商品 store、退款 store、购物车、积分、支付回调、联票券包等 mock 代码。C 端“我的订单/订单详情”只展示便民服务订单。

可复用设计：

| 模块 | 可复用点 |
| --- | --- |
| 商品订单状态 | 待支付、已支付、待发货、已发货、已完成、售后中等状态拆分 |
| 履约方式 | 核销履约与物流履约分流 |
| 售后 | 退款申请、审核、退款失败重试、退款进度提示 |
| 支付回调 | 签名校验、重复回调拦截、成功/失败分支 |

后续复现建议：

1. 该能力应归入商城/CRMEB 需求，不再和便民服务订单共用 C 端订单页。
2. 如确实要在古城游小程序聚合展示商城订单，应通过商城接口只读聚合，不在平台侧重建商品订单状态机。
3. 老链接或历史通知携带商城订单 id 时，不在当前 demo 页面给出“转交商城后台”的敷衍提示；应由独立商城需求设计可追溯的跳转、登录态、订单存在性校验和异常兜底。
4. 商城订单、门票核销、物流、售后、退款都应有明确责任系统、接口契约和验收标准后再恢复。

## 3. 商城派单（独立需求）

当前状态：已删除商城核销派单、商城服务人员、SKU 派单配置相关运行代码。

可复用设计：

| 模块 | 可复用点 |
| --- | --- |
| 派单规则 | 按商品/SKU、预约时段、服务人员能力匹配 |
| 任务流转 | 核销后生成服务任务、服务人员接单、完成回传 |
| 异常处理 | 超时未接单、人工改派、任务取消 |

后续复现以 [商城派单功能独立需求文档](/Users/lzz/Downloads/丽江古城游/demo/docs/商城派单功能独立需求文档.md) 为准。便民服务派单与商城派单必须保持两套业务口径、两套人员范围、两套验收标准。

## 4. 文化院落集章/成就玩法（移出本期）

当前状态：已删除 C 端“我的成就”、Desktop“成就管理”和本地成就 mock 数据。文化院落本期只保留“院落列表/详情/去打卡/我的打卡/Desktop 打卡记录”。

可复用设计：

| 模块 | 可复用点 |
| --- | --- |
| 电子集章 | 每个院落对应一枚章，打卡后点亮；适合节庆活动或线路推广 |
| 成就卡片 | 首次打卡、连续打卡、多院落打卡、夜游/周末等条件配置 |
| 活动传播 | 生成分享海报、完成证书、纪念章展示 |
| 后台配置 | 成就名称、达成条件、奖励文案、上下线开关 |

后续复现建议：

1. 先确认客户是否真的要做“活动运营玩法”，不要把简单打卡默认扩展成成就系统。
2. 如果做，建议单独立为“文化院落电子护照/集章活动”，与基础打卡分层。
3. C 端入口可以从“我的打卡”延伸，不要直接占据文化院落主流程。
4. 奖励应优先使用虚拟纪念章、电子证书、分享海报；如涉及优惠券或实物兑换，需要纳入商城/财务/核销链路。
5. Desktop 管理应只在活动确认后恢复，包含规则配置、记录统计、奖品核销和活动上下线。

## 5. 联票套餐、优惠券、积分与购物车（移出本期）

当前状态：已删除本地券包、优惠券、积分和购物车 mock。文化院落本期不承接交易奖励；如未来集章活动需要发券，应进入商城交易与核销需求。

可复用设计：

| 模块 | 可复用点 |
| --- | --- |
| 联票套餐 | 固定套餐/可选套餐、拆分结算、单码核销 |
| 优惠券 | 用户领券、发券、使用限制、过期状态 |
| 积分 | 积分冻结、过期、抵扣、退款回滚 |
| 购物车 | SKU 数量合并、勾选结算、金额重算 |

后续复现建议：

1. 与商城交易需求一起恢复，避免在本期便民服务小程序中形成半截交易链路。
2. 若文化院落集章活动未来需要发券，应明确优惠券来源、适用商城范围、核销系统和过期规则。
3. 联票套餐涉及自营/第三方结算拆分，需和商城结算、财务对账一起设计。

## 7. 酒吧附属信息（并入酒吧，不再单列菜单）

当前状态：`酒水单`、`歌手节目`、`商品类型` 不再作为 Desktop 一级菜单展示；历史链接统一收口到 `游客服务 > 酒吧`。本期 demo 通过酒吧主档案展示和维护基础附属信息，如酒吧类型、卡座数量、散座数量、表演开始时间、歌手节目时段。

可复用设计：

| 模块 | 可复用点 |
| --- | --- |
| 酒水单 | 以酒吧为归属维护商品清单、图片、分类、描述 |
| 歌手节目 | 以酒吧为归属维护驻唱/演出时段、歌手信息、节目说明 |
| 商品类型 | 作为酒吧商品的分类配置，支持酒水/节目等不同类型 |

后续复现建议：

1. 如后续客户确认要展开酒吧特色运营，可在酒吧详情内增加“酒水单 / 歌手节目 / 商品类型”三个页签，不建议再恢复为独立一级菜单。
2. 数据模型应继续以 `barId` 作为归属主键，避免和普通购物、餐饮商品混用。
3. 若 C 端需要单独展示酒吧节目单或酒水推荐，再补对应前台展示页，但后台入口仍建议挂在酒吧主档案下。

当前状态：Desktop 中原有的 `资讯文章` 菜单与本地 mock 文章数据已移除。内容管理按老文档口径保留 `景区资讯 / 古城攻略 / 服务中心 / 政策法规 / 保护指南 / 办事流程 / 古城维护费 / 文化古城 / 文化期刊 / 图片标识共享库 / 古城视频 / 精选线路 / 推荐线路`。

可复用设计：

| 模块 | 可复用点 |
| --- | --- |
| 文章表单 | 标题、分类、正文、封面、发布时间、状态管理 |
| 文章列表 | 标题检索、分类筛选、发布状态管理 |
| 文章详情 | 富文本内容与封面预览 |

后续复现建议：

1. 若后续确实需要“综合文章库”，建议先明确它与 `景区资讯`、`服务中心`、`政策法规` 的边界，再决定是否恢复独立菜单。
2. 在边界未明确前，不建议保留一个与老文档不一致、且与已有内容管理栏目高度重叠的 `资讯文章` 模块。
