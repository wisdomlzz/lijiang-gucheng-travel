# 架构设计原则与功能归属决议

## 一、设计原则

**目标真实架构，不屈服于临时需求。**

- 功能归属按业务域的**本质边界**划分，而非按当前 Demo 的交互形态
- 临时拼凑的能力（如 0 元购票冒充"院落预约"）不配独立成 feature
- 商城/交易系统是基础设施层，不是业务 feature — 它支撑多个业务域，而非自身成为一个"功能"
- feature 之间不 import，业务域边界即代码边界

> 架构是长期的，需求是流动的。用架构承载本质，用接口适配变化。

## 二、功能归属决议：booking × courtyard

### 背景

当前 `features/booking/` 的 `booking-store` 实现了"选时段 → 生成预约码 → 核销 → 我的预约"流程。这个流程被标记为"院落预约"，但实际上：

- **它是 0 元购票流程的简化模拟** — 选时段、生成凭证、核销，本质是订单/交易
- **它与商城功能完全同构** — 如果商城系统建好，这个流程就是商城订单的一个子场景
- **它没有独立的业务模型** — 不管理库存、不管理价格、不管理支付，只是一个状态机

### 结论：booking 不独立成 feature

```
❌ features/booking/        ← 删除，不是独立业务域
   ├── store/
   │   ├── booking-store.ts  ← 这是商城交易的简化占位
   │   └── index.ts
```

它的真实归属是未来的商城/交易系统（`platform/trade` 或 `features/trade`），但当前不创建该 feature——避免造一个空壳。

### 操作：文化院落增加预约跳转

文化院落详情页（`features/courtyard/`）中放置**预约入口按钮**，点击后：

```
当前状态：  跳转提示 "预约功能开发中"
未来状态：  跳转到商城/交易系统的购票页（届时由 trade feature 接管）
```

具体页面不做——只有跳转点和占位提示。这保持了"文化院落展示"的信息流纯净，又不堵死未来的交易流程接入。

### 影响范围

| 项 | 操作 |
|----|------|
| `features/booking/` | 🔴 整体删除 |
| 现有引用 `booking-store` 的页面（CourtyardBookingPage, MyBookingsPage） | 🔴 随 booking 删除 |
| `features/courtyard/` | 🟢 详情页加预约跳转按钮（仅占位，无页面） |
| 商城交易系统 | ⏳ 未来实现，届时接管预约流程 |

## 三、延伸：content 拆 5 个 feature

按同一原则，当前 `features/content/` 打包了 5 个不相关的业务域，一并拆解：

| 原模块 | 独立 feature | 理由 |
|--------|-------------|------|
| `merchant-store` | **merchant** 商户 | 购在古城（商户列表/详情/服务），有独立 C 端页面集群 |
| `courtyard-store` | **courtyard** 文化院落 | 院落展示+VR，信息流独立 |
| `poi-store` | **map** 导览地图 | 地图 POI 数据，唯一驱动 MapPage |
| `news-store` | **news** 古城资讯 | 古城资讯+便民信息，通知类内容 |
| `guide-store` | **routes** 精选路线 | 路线导览，独立的列表+详情+预览交互 |
