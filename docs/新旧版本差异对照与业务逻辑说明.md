# 新旧版本差异对照与业务逻辑说明

> 口径说明：`markdown_exports` 为老版本资料和历史功能基线；`demo` 为新版三端演示。新版 demo 重点展示本期业务闭环，部分老版本已有能力采用“复用原小程序/原后台/外部系统”的方式保留入口，因此没有重新制作完整 demo 页面。
>
> 商城商品订单、商品核销、物流、退款、商城派单、商城服务人员、SKU 派单配置等商城交易与履约需求，统一交由另一个独立需求承接。本期 demo 保留供应商入驻、商城管理后台入口，以及购在古城/附近/酒吧/餐饮/住宿等商家信息维护能力。

## 一、总体变化

| 维度 | 老版本（markdown_exports） | 新版本（demo） | 变化说明 |
|---|---|---|---|
| 产品形态 | 小程序 + 管理后台 + 官网/商城跳转 | C端游客小程序 + B端便民服务人员小程序 + Desktop管理后台 | 新增 B 端便民服务履约角色，三端围绕便民服务闭环 |
| 业务重点 | 景区信息展示、商家/便民信息发布、地图导航、后台内容维护 | 便民服务订单、自动派单、现场报价、投诉处置、诚信评分、遗产数据管理、商家信息维护、供应商入驻、文化院落/随手拍互动 | 从“展示型平台”升级为“运营型平台”；商城商品交易与商城派单另立需求 |
| 交易方式 | 多数交易跳转有赞/商城/外部系统 | 本期 demo 只描述便民服务履约链路；商城商品交易通过后续 SSO 对接商城后台承接 | 避免把商城核销、物流、退款、商城派单纳入本次验收 |
| 后台定位 | 内容、POI、商家、游客服务基础数据维护 | 运营驾驶舱 + 便民服务调度 + 服务人员/片区/投诉/评价/遗产管理 + 信息维护 + 供应商入驻审核 | 后台从数据录入扩展到运营处置 |
| 老功能处理 | 老版文档内完整描述 | 一部分做新版页面，一部分提示“复用原小程序/原后台”，商城商品交易与商城派单另出需求 | 不能按老版文档逐项要求 demo 页面 |

## 五、新版主要业务逻辑变化

### 1. 便民服务从“信息入口”变为“订单履约闭环”

新版一键服务支持 6 类服务：生活垃圾清运、建筑垃圾清运、布草配送、送水服务、行李搬运、送货服务。

核心流程：

1. C端提交服务订单：选择服务、地址、时间、描述、现场照片。
2. 系统自动派单：点对点服务按距离和接单量分配；片区服务按片区/服务站分配。
3. 派单失败进入人工处理：后台可查看派单流水并手动指派。
4. B端服务人员接单，到场后报价。
5. 用户线上支付或线下支付；超时/争议进入异常处理。
6. 服务人员完工拍照，用户确认完成，订单结束。
7. 用户评价后影响服务人员诚信分。

关键状态：

| 状态 | 含义 |
|---|---|
| S10 | 已下单/待进入派单 |
| A10 | 待派单 |
| A20 | 已指派 |
| A30 | 已接单 |
| A35 | 已核价 |
| A38 | 协商中 |
| A40 | 已收款 |
| S48 | 服务中 |
| S55 | 完工待确认 |
| S40 | 已完成 |
| S50 | 已取消 |
| R80 | 取消审批中 |
| S90 | 待人工处理 |

### 2. 商家信息维护与供应商入驻纳入本期，商城交易与商城派单另出需求

本期保留两类能力：

| 本期保留 | 说明 |
|---|---|
| 购在古城、附近、酒吧/餐饮/住宿 | 作为商家/POI 信息展示与后台信息维护保留 |
| 供应商入驻 | C端提交、后台审核，审核通过后为后续商城后台 SSO 做准备 |
| 商城管理后台入口 | 后续平台和商城对接后，供应商单点登录到商城后台管理自己的商品、订单等 |

商家生态后台字段口径按老文档拆分，不再用一套泛化“商家”字段覆盖：

| 模块 | 当前对齐字段 |
|---|---|
| 酒店（住宿） | 酒店名称、酒店位置、酒店电话、经纬度、360 全景链接、酒店描述、图片，按酒店名称/来源/审核状态检索 |
| 餐饮 | 餐厅名称、餐厅电话、餐厅位置、经纬度、360 全景链接、餐厅介绍、图片，按餐厅名称/来源/审核状态检索 |
| 购物 | 商家名称、地址、经纬度、电话、资质、商品详情图片、描述（富文本），按商家名称/来源/审核状态检索 |
| 酒吧 | 酒吧名称、关联用户、酒吧类型、酒吧位置、经纬度、卡座数量、散座数量、表演开始时间、歌手节目时段、营业时间、联系电话、描述图片，按酒吧名称/酒吧类型/表演开始时间/开业时间检索 |

商城交易与商城派单单独出需求，不在本期 demo 展示：

| 独立需求范围 | 本期处理方式 |
|---|---|
| 商品发布、商品订单、核销码、物流、退款 | 由商城后台承接 |
| 商城核销派单、商城服务人员、SKU 派单配置 | 本期 demo 已删除，另出《商城派单功能需求文档》 |
| 门票、联票、周边游等商城交易 | 保留外部入口，后续由商城需求承接 |

C端小程序订单相关页面当前只保留便民服务订单：游客可从一键服务下单、查看便民订单列表、支付报价、确认完成和评价；商城商品订单不在 C端 demo 订单页中展示。

### 3. 投诉从表单提交变为可处理状态流

老版本一键投诉主要是游客填写投诉内容提交。新版增加后台处置状态：

| 状态 | 含义 |
|---|---|
| C10 | 已提交 |
| C20 | 已受理 |
| C30 | 处理中 |
| C40 | 已处理 |
| C50 | 已完结 |

后台可按投诉对象、关联服务/订单、状态处理；涉及商城商品订单的投诉后续由商城后台对接承接。

### 4. 遗产保护从资料表升级为前后台管理模块

老版本遗产保护资料来自可视化管理梳理表，重点是道路、水系、古井、古桥、古树名木、代表性民居、公共构筑物、人文环境、文保单位等资料整理。

新版 demo 将其中一部分转为：

| C端 | Desktop |
|---|---|
| 遗产知识分类浏览、详情页、离我最近 | 遗产分类列表、详情、新建、编辑、删除 |

这属于新版的明显新增方向，不再只是旧后台 POI 数据维护。

### 5. 文化院落新增游客打卡闭环

老版本文化院落主要是列表、详情、导航、VR、语音、分享。新版保留这些展示能力，并增加轻量打卡：

| 新增能力 | 说明 |
|---|---|
| 院落打卡 | 游客上传照片和定位 |
| 我的打卡 | 游客查看院落打卡记录 |
| 后台打卡记录 | Desktop `photo-records` 查看记录与状态 |

> 说明：市面上的“集章/徽章/成就”常用于活动营销或二次传播，但客户当前只要求打卡功能，因此当前版本不把成就管理作为主流程。

### 6. 诚信评分成为服务治理逻辑

新版增加服务人员诚信评分：

| 触发 | 影响 |
|---|---|
| 好评 | 加分 |
| 投诉成立 | 扣分，低于阈值进入观察期 |
| 多次拒单 | 扣分 |
| 管理员调整 | 后台留调整记录 |

老版本没有这一套服务治理逻辑。

## 六、明确不应按“缺失”判断的复用部分

以下能力在老版本已有完整实现，新版 demo 当前主要保留入口、提示或外部跳转，不建议按“新版未完成”直接判定：

## 七、当前 demo 需要特别说明的实现口径

1. 新版 demo 是本地 mock 数据驱动，不是接真实接口。
2. C 端地图导览、线路、便民信息、资讯和 VR 已恢复实际路由，仍按旧版成熟小程序口径复用，不纳入新版便民服务状态机。
3. Desktop 中角色管理、网站管理、便民服务审核、访问统计、广告位维护和官网内容维护已由“提示复用”推进为可演示页面；后续继续按老文档截图逐项校验字段细节。
4. 供应商入驻属于本期需求；后续平台和商城对接后，审核通过的供应商通过 SSO 进入商城后台管理自己的商品、订单等。
5. 首页“购在古城”当前指向 `/c/merchants`，购在古城/附近/酒吧/餐饮/住宿按商家和 POI 信息维护保留，不和商城派单混为一谈。
6. 商城核销派单、商城服务人员、SKU 派单配置已经从本期 demo 入口和源码中删除，另出独立需求文档。
7. C端“我的订单/订单详情”只展示便民服务订单；历史商城订单、门票订单、物流/售后订单不在当前 demo 内做提示性承接，后续需由独立商城需求设计可追溯跳转、登录态校验和异常兜底。
8. 首页底部资讯区已收敛为“古城资讯与便民信息”，不再展示志愿服务招募。
9. 志愿服务功能当前暂时搁置具体业务流程，仅保留首页宫格占位和 `/c/volunteer` 需求待澄明页面；活动报名、审核、时长记录和后台 CRUD 暂不纳入当前 demo。
10. 首页“随手拍”配置路由写作 `/c/snapshot`，点击时在首页代码中被特殊处理为 `/c/photo-report`。

## 八、建议对外表述

新版 demo 的验收口径建议写成：

> 新版 demo 重点演示丽江古城游从“信息展示型小程序”向“服务运营型平台”的升级：本期核心闭环是 C端便民服务下单、C端便民服务订单管理、B端便民服务人员接单履约、Desktop 后台便民服务派单调度、投诉处置、评价/诚信评分，以及遗产管理、文化院落打卡、随手拍、供应商入驻审核、购在古城/附近/酒吧餐饮住宿等商家信息维护。商城商品订单、核销、物流、退款和商城派单由后续商城后台及独立商城派单需求承接；本期只保留商城管理后台入口和供应商入驻，不再展示商城派单功能。首页底部资讯收敛为古城资讯与便民信息聚合，志愿服务保留占位入口但业务需求待澄明。老版本已成熟的地图导览、资讯线路、便民信息、VR全景、部分游客服务和官网后台栏目，按复用原小程序/原管理后台处理，因此不重复制作完整 demo 页面。
