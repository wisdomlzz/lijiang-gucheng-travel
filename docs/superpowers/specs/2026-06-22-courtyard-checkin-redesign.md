## 2. 数据模型决策

### 2.1 打卡防刷规则

**规则**：同一个人（userId）在同一天内对一个院落（courtyardId）最多只允许打卡 1 次。

```
每日限制 = userId + courtyardId + date（精度到日）
重复打卡 → 提示"今日已打卡"，不创建记录
隔天可重新打卡
```

**理由**：
- 理解成本最低（"今天已经打过卡了"）
- 数据可信——榜单展示的是"有 N 人次到访（逐日独立）"
- 防止刷量行为破坏统计真实性

**不受限制的场景**：
- 不同 userId → 独立计数
- 不同 courtyardId → 独立计数
- 同人不同日 → 独立计数

### 2.2 榜单统计口径

| 字段 | 值 |
| --- | --- |
| 统计维度 | **打卡次数（人次）** |
| 每日去重 | ✅ 同人同院同日仅算 1 次 |
| 数据来源 | `checkin-store.checkins` 按 `courtyardId` 分组 |
| 排序 | 按计数降序 |
| 默认展示 | Top 10，可展开全部 |

**注意**：口径为"人次"而非"人数"。同一个游客在不同日期来打卡 3 次，榜单记 3 次。

### 4.2 三区域说明

#### 区域 1：KPI 统计行
- 总打卡次数 — 总记录数（去重后）
- 参与游客 — 去重 userId 计数
- 已打卡院落数 — 有打卡记录的 courtyardId 去重计数
- 样式：圆角卡片，紧凑文字行，不带大号数字卡片

#### 区域 2：各院落打卡排行榜
- 水平条形图，每行包含：排名序号 | 院落名 | 进度条 | 次数 | 占比
- 第1名蓝色高亮渐变，其余灰色渐变
- 默认展示 Top 10，>10 个时显示"展开全部"按钮
- **纯展示**，无点击筛选联动
- 排行榜数据来自 `checkin-store` 按 `courtyardId` 分组聚合

#### 区域 3：打卡明细表格
- 列：院落名 | 用户 | 位置 | 打卡时间 | 查看(按钮)
- 筛选：游客姓名模糊搜索 + 开始/结束日期
- 分页：10 条/页

### 4.3 与桌面端其他页面的关系

- 工作台（Workbench）的"打卡记录"快捷卡片 → 跳转到该页面
- 详情页（show.tsx）→ 显示单条打卡的照片和详细信息

## 6. 打卡限流 API 设计

### 6.1 新增校验逻辑

```typescript
// checkin-store.ts 新增方法
canCheckin(userId: string, courtyardId: string): { allowed: boolean; reason?: string }

// 实现：
function canCheckin(userId, courtyardId) {
  const today = new Date().toISOString().slice(0, 10) // "2026-05-18"
  const existing = checkins.find(
    c => c.userId === userId 
      && c.courtyardId === courtyardId 
      && c.createdAt.startsWith(today)
  )
  if (existing) return { allowed: false, reason: "今日已打卡" }
  return { allowed: true }
}
```

### 6.2 C端详情页变更

- `handleCheckin` 调用前先走 `canCheckin` 校验
- 不允许时 → 按钮置灰，显示"今日已打卡"
- 打卡成功弹窗文案不变

### 6.3 注意事项

- `seed` 数据的 `createdAt` 是字符串，日期判断直接按字符串前缀 `startsWith` 匹配 "2026-05-"
- 种子数据的日期分布在 05-12 ~ 05-29，不影响当日新打卡判断（当日是 06-18）
