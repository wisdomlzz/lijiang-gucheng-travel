# 购在古城 — 古城商户业务体系设计

## 1. 业务定位

**"购在古城" ≠ 电子商城。**

它是：

> **丽江古城的店铺信息平台** — 让游客发现古城里有什么店、在哪、有什么特色；
> **商户身份管理系统** — 让店主认领自己的店铺，维护信息，平台审核保障准确性。

它与**线上商城供应商体系**是正交关系——两者是**独立子系统**：

| 维度 | 古城商户（子系统一） | 线上商城供应商（子系统二） |
|------|---------|--------|
| 核心标识 | "我在古城有实体店" | "我能向商城提供商品/服务" |
| 认证依据 | 实体店铺位置、经营权 | 经营资质、供货能力 |
| 面向 | 游客发现（C端购在古城） | 商城供货（线上商城） |
| 独立性 | 可独立存在 | 可独立存在 |
| 重叠 | 同一个用户可同时是古城商户 + 线上商城供应商 | |

> **约束：** "购在古城"（MerchantListPage）代码不重构，只做增量优化（如展示认领状态标记）。

## 2. 核心概念模型

### 2.1 最终真相：古城里有实体店铺

一切围绕**古城中真实存在的实体店铺**展开。一家店铺在系统中只需要被创建一次，之后围绕它进行各种操作。

### 2.2 店铺的两种创建路径

```
                    ┌──────────────────────┐
                    │  古城里的实体店铺      │
                    │  (物理存在，独立于系统) │
                    └──────────┬───────────┘
                               │
            ┌──────────────────┼──────────────────┐
            ▼                  ▼                   ▼
    ┌──────────────┐   ┌──────────────┐   ┌──────────────┐
    │ 平台导入     │   │ 平台已导入    │   │ 商家认领/    │
    │ (批量预载)   │   │ (可被认领)   │   │ 新建申请     │
    └──────┬───────┘   └──────┬───────┘   └──────┬───────┘
           │                  │                   │
           ▼                  ▼                   ▼
    ┌──────────────┐   ┌──────────────┐   ┌──────────────┐
    │ 店铺入库     │   │ 认领审核     │   │ 入驻审核     │
    │ claimStatus  │   │ (绑定身份)  │   │ (创建+绑定)  │
    │ = unclaimed  │   └──────┬───────┘   └──────┬───────┘
    └──────┬───────┘          │                   │
           │                  └────────┬──────────┘
           │                           ▼
           │                  ┌──────────────────┐
           └─────────────────►│  已验证的店主     │
                              │  (可维护店铺信息) │
                              └──────────────────┘
                                       │
                              ┌────────┴────────┐
                              ▼                  ▼
                       ┌────────────┐    ┌────────────┐
                       │ 购在古城展示 │    │ 信息变更审核 │
                       │ (游客浏览)  │    │ (每次修改审) │
                       └────────────┘    └────────────┘
```

### 2.3 店铺状态机

```typescript
type ClaimStatus = "unclaimed" | "pending" | "claimed"
type ReviewStatus = "通过" | "不通过" | "待审核"
```

| ClaimStatus | 含义 | 展示 | 可操作 |
|-------------|------|------|--------|
| `unclaimed` | 平台导入，无人认领 | 基础信息，标记"待认领" | 商家可发起认领 |
| `pending` | 认领审核中 | 基础信息，标记"认领审核中" | 认领按钮置灰 |
| `claimed` | 已绑定店主 | 完整信息 | 店主可维护，每次修改走审核 |

### 4.2 认领/入驻统一入口（核心页面）

用户进入后只需要回答一个问题：**"您的店叫什么名字？"**

```
┌─────────────────────────────────┐
│  认领或入驻店铺                   │

│                                 │

│                                 │
│  🏪 纳西印象餐厅               │
│     七一街                       │
│     [这就是我的店]              │
│                                 │
│  ──────────────────────────    │
│  没找到？[点此提交新店铺信息]     │
└─────────────────────────────────┘
```

#### 认领流程（匹配到已有店铺）
```
已有店铺详情 → "这就是我的店"
  → 填写认领信息（姓名、电话、关系证明描述）
  → 提交审核
  → 后台审核（确认申请人与店铺的关系）
  → 通过：店铺 claimStatus → claimed，用户获得管理权限
  → 驳回：告知原因
```

#### 新建流程（未匹配到已有店铺）
```
没找到您的店？
  → 填写店铺信息（店名、类型、地址、电话、简介、营业时间）
  → 填写认领信息（姓名、电话、关系证明描述）
  → 提交审核
  → 后台审核（确认店铺真实存在 + 确认关系）
  → 通过：创建 Merchant + 用户获得管理权限
  → 驳回：告知原因
```

### 4.4 购在古城（双端展示）

"购在古城"在游客端和商户端都存在，展示逻辑不同：

| 端 | 用户 | 展示内容 | 认领标签 |
|---|------|---------|---------|
| 游客端（C端） | 普通游客 | 全部店铺信息 | ❌ 不显示 |
| 商户端（C端商户服务→购在古城） | 已登录商户 | 全部店铺信息 + 认领状态 | ✅ 显示"待认领/审核中/已认领" |

**实现方式：**
- 现有 MerchantListPage 根据当前用户 `roles` 条件渲染认领状态标签
- 游客端：保持现有逻辑，不显示任何认领相关的 UI
- 商户端：多渲染 claimStatus 标记，未认领店铺显示"认领此店"按钮
- 此为**增量优化**，不重构 MerchantListPage 核心逻辑

## 6. 线上商城供应商入驻（C端）

> **命名说明：** 明确改为"线上商城供应商"，以区别于古城商户体系。它是独立的子系统二。

线上商城供应商入驻需在 C 端也提供入口（当前仅桌面端有 SupplierEntryDesktop）。

在"商户服务页-供应商合作"下增加 C 端入驻表单：
- 复用现有 SupplierApplication 数据模型
- 表单内容：公司/店铺名称、联系人、电话、经营类型、地址、营业执照、经营范围简介
- C 端提交 → 后台 SupplierApplicationsList 审核
- 后台审核通过 → 用户在桌面端拥有供应商权限

### 两个子系统的桥接

同一个用户可同时是古城商户 + 线上商城供应商：

```
用户账户
├── 古城商户身份 ← 认领/入驻审核通过后获得
│   ├── 管理店铺信息（我的店铺）
│   └── 购在古城展示
│
└── 线上商城供应商身份 ← 供应商入驻审核通过后获得
    ├── 管理商品/服务（桌面端）
    └── 商城后台
```

在"我的店铺"和"商户服务页"中，增加"申请成为线上商城供应商"的入口链接。
