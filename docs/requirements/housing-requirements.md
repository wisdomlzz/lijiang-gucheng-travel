## 一、产品定位与边界

### 1.1 我们在做什么

公房服务模块提供丽江古城国有公房（公有产权房屋）的信息查询与管理能力。公房是古城内特殊的房产类型，包含沿街铺面、院落等多种形态。本模块面向游客/商户提供公房信息的浏览和详情查询，同时为平台管理员提供公房数据的维护管理功能。

**核心能力：**
- C端游客/商户：浏览公房列表、查看详情（基础信息、合同信息、租赁信息）、全景预览（模拟）—— 仅信息展示，不含交易流程
- 桌面端管理员：公房数据的增删改查管理

### 1.2 明确不做的（模块边界）

以下功能在本模块范围内不实现：
- ❌ 租赁申请/签约流程（仅展示合同信息，不含签约动作）
- ❌ 房租在线支付
- ❌ 公房地图定位标记（仅展示坐标文本）
- ❌ 真实图片上传（目前为渐变占位图）
- ❌ 公房报修/投诉
- ❌ B端（便民服务人员端）相关页面
- ❌ 公房统计分析/报表

## 三、核心业务流程

### 3.1 C端公房查询流程

```
用户打开「公房信息」入口（首页宫格或直接导航到 /c/housing）
    ↓
公房列表页加载：展示∑全部/出租/未出租统计 + 卡片列表
    ↓
用户通过 Tab 筛选（全部 / 出租 / 未出租）
用户通过搜索框按名称/地址搜索
    ↓
列表分页加载（每页 10 条，点击"加载更多"追加）
    ↓
点击公房卡片进入详情页
    ↓
详情页展示：
  ├─ 照片轮播（渐变占位图，左右切换 + 页码指示器）
  ├─ 房源摘要：名称、地址、状态标签、面积/商业类型/结构
  ├─ 基础信息：出租状态、门牌号、地图坐标、房屋状态、房产证地址、商业类型、面积、结构、用途 等
  ├─ 合同信息：租期、签订日期、月租金、押金、合同状态、商铺名称 等
  └─ 租赁信息：应缴金额、缴费状态、实缴金额
    ↓
（可选）支持全景标识的公房可进入全景漫游模式（模拟进度条播放）
    ↓
退出详情页返回列表
```

### 3.2 桌面端数据管理流程

```
管理员登录桌面端 → 内容管理 → 公房信息 Tab
    ↓
表格展示所有公房（支持搜索 + 分页）
    ↓
操作：
  ├─ 新增公房：填写名称、地址、状态、区域、摘要信息 → 保存
  ├─ 编辑公房：修改已有字段 → 保存
  └─ 删除公房：确认后删除数据
```

## 五、核心数据模型

### 5.1 数据库表 `content_housing`

```sql
CREATE TABLE IF NOT EXISTS content_housing (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  name TEXT NOT NULL,                           -- 公房名称，如"光义街9号公房"
  addr TEXT NOT NULL,                           -- 完整地址
  status TEXT DEFAULT 'rented',                 -- 出租状态：rented / idle
  statusText TEXT DEFAULT '出租',               -- 中文显示文本
  area TEXT DEFAULT 'gucheng',                  -- 区域编码
  areaName TEXT DEFAULT '古城区',               -- 区域名称
  meta TEXT DEFAULT '[]',                       -- JSON 数组：[面积, 商业类型, 建筑结构]
  createdAt TEXT NOT NULL DEFAULT (datetime('now')),
  updatedAt TEXT NOT NULL DEFAULT (datetime('now'))
);
```

### 5.2 前端类型定义

```typescript
// housing-store.ts
interface HousingItem {
  id: number
  name: string           // 公房名称，如"光义街9号公房"
  addr: string           // 完整地址
  status: "rented" | "idle"  // 出租/未出租
  statusText: string     // 中文显示文本
  area: string           // 区域编码
  areaName: string       // 区域名称
  meta: string[]         // 摘要标签：[面积, 商业类型, 结构]
}

// HousingPage.tsx 中展示用扩展模型（仅前端静态数据使用）
interface House extends HousingItem {
  hasPano: boolean                 // 是否有全景
  photos: { bg: string }[]        // 背景渐变图（占位）
  basic: Field[]                  // 基础信息字段列表
  contract: Field[]               // 合同信息字段列表
  lease: Field[]                  // 租赁信息字段列表
}

interface Field {
  label: string
  value: string
  cls?: string        // "price" 标记为价格类（蓝色高亮）
  badge?: string      // 对应的 badge 样式 key
}
```

### 5.3 数据关系说明

- **后端存储模型**（表字段）：仅存储核心字段（name, addr, status, area, meta），不存储照片、不拆分合同/租赁字段。详情页展示的完整信息（basic/contract/lease 分组）由前端静态数据补充。
- **前端呈现模型**（`House` 类型）：在页面内硬编码了 5 条完整真实数据 + 50 条随机生成数据，用于 C 端展示。照片为渐变背景占位，无真实图片。
- **桌面端管理**：通过 store 调用后端 API 进行 CRUD，数据模型使用 `HousingItem` 简版结构。新增/编辑时仅录入名称、地址、状态、区域、meta 标签。

## 七、技术架构说明

### 7.1 架构分层

```
┌─────────────────────────────┐
│  C端 HousingPage            │ ← 页面静态数据（5+50 条），含全景模拟
│  (src/features/housing/     │
│   c-end/pages/)             │
├─────────────────────────────┤
│  桌面端 HousingManageContent │ ← CRUD 表格，通过 store 调 API
│  (src/desktop/pages/gates/  │
│   content/)                 │
├─────────────────────────────┤
│  useHousingStore (zustand)  │ ← 同步器模式（syncAction），管理 houses 状态
│  (src/features/housing/     │
│   store/)                   │
├─────────────────────────────┤
│  contentApi.housing         │ ← API 客户端（list/create/update/remove）
│  (src/api/client.ts)        │
├─────────────────────────────┤
│  /api/content/housing (CRUD)│ ← 通用 CRUD 路由
│  (server/routes/content.js) │
├─────────────────────────────┤
│  SQLite: content_housing    │ ← 持久化表
│  (server/db/schema.sql)     │
└─────────────────────────────┘
```

### 7.2 关键设计决策

1. **C端与桌面端数据源分离**：C端 HousingPage 使用页面内静态数据（5条真实+50条生成）进行展示，不依赖后端 API。桌面端 HousingManageContent 通过 zustand store 调用后端 API 进行 CRUD。两者数据不同步。

2. **详情页模型扁平化**：数据库仅存储核心字段，不拆分为基础/合同/租赁三个子集合。详情页的三个信息分组由前端在代码中静态定义字段列表，按 `basic` / `contract` / `lease` 分组渲染。

3. **全景漫游为模拟效果**：全景功能不涉及真实 720° 全景资源或 Three.js 渲染，仅通过深色背景 + 进度条播放器模拟全景浏览体验。

4. **数据加载**：应用初始化时通过 hydrate.ts 调用 `api.list("content/housing", { pageSize: 200 })` 加载所有公房数据到 zustand store，供桌面端使用。`syncAction` 装饰器确保 store 状态与 API 响应一致。
