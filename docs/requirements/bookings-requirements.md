# 文化院落预约模块 — 产品需求文档

> **文档版本**：v1.0
> **更新日期**：2026-07-07
> **产品定位**：丽江古城文化院落预约功能，支持 C 端游客在线预约参观时段 + 桌面端管理员管理预约 + 核销码线下验票
> **关联模块**：打卡签到（checkin）、内容管理（courtyard-store）、商户审核（merchant-review）
> **实现状态**：已有 C 端 MVF 实现（选时段 → 预约码 → 核销 → 我的预约），缺失桌面端管理、容量强约束、超时自动过期

---

## 一、定位与边界

### 1.1 我们在做什么

文化院落预约功能是丽江古城旅游服务平台面向 C 端游客的独立预约通道。游客可以浏览古城内开放的付费/免费文化院落，在线选择日期和时段预约参观，到院落后出示核销码完成核销。

**核心目标：**

- ✅ 验证「浏览院落 → 选时段 → 预约生成 → 线下核销」这条主线能不能跑通
- ✅ 验证游客是否愿意提前规划参观行程
- ✅ 攒第一批预约数据和到访数据
- ✅ 为院落人流管理提供数据基础

### 1.2 MVP 原则

| 优先级 | 原则 | 说明 |
|--------|------|------|
| 必须 | 主流程必须闭环 | 浏览→预约→核销→记录，每一步不能断 |
| 必须 | 时段冲突检测 | 同一院落同一天同时段禁止重复预约 |
| 必须 | 核销码唯一可验 | 每个预约生成唯一核销码，支持线下出示+管理员核销 |
| 可以简化 | 容量管理 | 每时段固定上限 20 人，先到先得，不做动态调整 |
| 可以简化 | 取消流程 | 直接取消，不设扣费规则（区别于便民服务取消审批） |
| 可以简化 | VR 游览 | 静态图片全屏展示，不做 360 全景 |
| 以后做 | 平台端容量配置 | 管理员后台可调整每院落每时段容量 |
| 以后做 | 批量预约（团队票） | 多人一次性预约多时段 |
| 以后做 | 预约提醒推送 | 预约日前一天系统消息提醒 |
| 以后做 | 预约转赠 | 核销码可在用户间转移 |

### 1.3 明确不做的（MVP 边界）

以下功能 **1.0 绝对不做**，避免范围蔓延：

- ❌ 支付/收费预约（所有院落预约免费）
- ❌ 预约转赠/代预约
- ❌ 多次预约插队（一个用户同日同时段只能一条预约）
- ❌ 管理员手动改预约（只能核销/取消）
- ❌ 取消扣费规则
- ❌ 预约日短信/推送提醒
- ❌ 未核销惩罚机制（如信用扣分）
- ❌ 团队/企业批量预约
- ❌ 真实 360 全景 VR（使用图片全屏模拟）
- ❌ 预约与入住/订单绑定

## 二、用户角色

### 2.1 两类角色

| 角色 | 端 | 核心诉求 |
|------|----|----------|
| **C 端游客** | 小程序（390×844） | 浏览文化院落、快速预约、到院后展示核销码、查看我的预约记录 |
| **平台管理员** | 桌面端后台 | 查看所有预约记录、按状态/院落/日期筛选、核销/取消预约 |

### 2.2 院落资源说明

| 类型 | 说明 | 是否需要预约 |
|------|------|-------------|
| 收费院落（如木府） | 需购票进入，独立售票体系 | MVP 阶段不处理 |
| 免费开放院落（如方国瑜故居） | 免费参观，需预约时段 | ✅ 需要 |
| 藏于巷道的微型院落 | 无人值守，扫码入内 | 以后接入 |

> MVP 阶段仅覆盖免费开放且支持预约的文化院落，收费院落由商户独立管理，不纳入预约系统。

## 三、核心业务流程

### 3.1 预约主流程（主线必须闭环）

```
用户进入文化院落列表页
    ↓
点击目标院落 → 进入详情页
    ↓
点击「预约参观」
    ↓
选择日期（未来 7 天） + 时段（5 个固定时段） + 人数（1-10）
    ↓
系统校验：该院落同日同时段已预约人数 + 当前人数 ≤ 20
    ↓
生成预约记录 + 6 位核销码
【status = pending】
    ↓
用户可查看我的预约记录（含核销码）
    ↓
到访院落 → 向工作人员出示核销码
    ↓
管理员在后台输入核销码 / C 端管理员扫码
【status → checked】
    ↓
流程结束
```

### 3.2 取消流程

```
用户在「我的预约」页面对 pending 状态的预约点击取消
    ↓
直接取消，无审批流程
【status → cancelled】
    ↓
释放该时段的容量占用
```

### 3.3 过期自动处理

```
预约日当天 23:59 仍未核销
    ↓
Server 定时任务扫描
【status → expired】
    ↓
释放该时段的容量占用（当日已过期，不必释放，仅用于统计）
```

### 3.4 核销流程（管理员视角）

```
管理员在桌面端预约管理页
    ↓
输入核销码 / 在列表中按状态筛选找到目标预约
    ↓
系统校验：
  ├─ 核销码有效且 status = pending → 核销成功 → status → checked
  └─ 核销码无效/已核销/已取消 → 提示失败原因
    ↓
更新 checkedAt 为当前时间
```

## 四、预约状态机设计

### 4.1 状态列表

| 状态码 | 状态名 | 说明 |
|--------|--------|------|
| pending | 待核销 | 预约成功，等待到院核销 |
| checked | 已核销 | 到院并核销成功，流程结束 |
| cancelled | 已取消 | 用户主动取消 |
| expired | 已过期 | 预约日结束未核销，自动过期 |

### 4.2 状态流转规则

> **注意**：所有状态变更应由 Server 端驱动，前端发请求触发。

| 源状态 | 目标状态 | 触发动作 | 触发方 |
|--------|----------|----------|--------|
| - | pending | createBooking | C 端用户 |
| pending | checked | checkBooking（核销验证） | 管理员（桌面端）/ C 端扫码 |
| pending | cancelled | cancelBooking | C 端用户 |
| pending | expired | autoExpire（定时任务） | Server 端 |
| checked | - | 不可再变更 | - |
| cancelled | - | 不可再变更 | - |
| expired | - | 不可再变更 | - |

### 4.3 状态与 UI 展示

| 状态 | C 端「我的预约」 | 桌面端管理列表 |
|------|------------------|----------------|
| pending | 显示核销码、可取消 | 标黄「待核销」、可操作核销 |
| checked | 显示核销时间、绿色 | 标绿「已核销」、只读 |
| cancelled | 置灰「已取消」、只读 | 标灰「已取消」、只读 |
| expired | 置灰「已过期」、只读 | 标灰「已过期」、只读 |

## 五、功能模块清单

### 5.1 P0 必须有（MVP 缺一不可）

#### C 端游客
- 文化院落列表浏览（分页加载，每页 6 条）
- 院落详情页（图文混排 + 基本信息 + 语音导览 + 打卡 + 预约入口）
- 选择日期（未来 7 天）
- 选择时段（5 个固定时段）
- 选择参观人数（1-10）
- 时段容量实时显示（固定 20 人上限）
- 提交预约 → 生成核销码
- 预约成功页（展示核销码 + 院落信息）
- 我的预约记录列表（按状态分类展示）
- 取消预约（仅 pending 状态）
- 核销码文本展示（到院出示给工作人员）

#### 桌面端管理员
- 预约管理列表（全部记录，支持按院落、日期、状态筛选）
- 核销码核销功能（输入核销码 → 核销 → 更新状态为 checked）
- 预约详情只读查看
- 手动取消预约（pending 状态）

#### 系统能力
- 核销码唯一生成（前缀 `YY` + 时间戳后 6 位 + 随机 2 位）
- 星期冲突检测（同日同时段 + 同一院落 + 非 cancelled 状态）
- 操作日志记录（预约创建、核销、取消、过期均记录）

### 5.2 P1 建议有（有空就做）

- 预约数据统计（每日/每周预约量、核销率、各院落热度）
- 管理员批量核销（连续扫码模式）
- 预约人数超限友好提示（展示剩余名额）
- 预约时段热度展示（各时段预约量趋势）

### 5.3 P2 以后做（远期规划）

- 预约消息推送（预约成功、预约日前一天）
- 管理员配置每院落每时段容量
- 管理员配置可预约日期范围（非固定 7 天）
- 付费预约接入（与商户售票系统对接）
- VR 真实 360 全景/三维漫游
- 核销码扫码核销（摄像头扫码，替代手动输入）
- 转赠/代预约
- 预约与信用分关联（恶意未核销扣分）

## 六、数据模型

### 6.1 预约表（bookings）

| 字段 | 类型 | 说明 |
|------|------|------|
| id | TEXT | 预约 ID |
| userId | TEXT | 用户 ID |
| courtyardId | TEXT | 院落 ID |
| courtyardName | TEXT | 院落名称（冗余，便于列表展示）|
| userName | TEXT | 用户姓名 |
| userPhone | TEXT | 用户手机号 |
| date | TEXT | 预约日期（YYYY-MM-DD） |
| timeSlot | TEXT | 时段字段（兼容，可能有值） |
| slot | TEXT | 时段（如 "09:00-10:00"） |
| visitors | INTEGER | 参观人数，默认 1 |
| code | TEXT | 核销码（唯一，YY + 时间戳后6位 + 随机2位） |
| status | TEXT | 状态（pending / checked / cancelled / expired）|
| checkedAt | TEXT | 核销时间 |
| createdAt | TEXT | 创建时间 |
| updatedAt | TEXT | 更新时间 |

**当前表结构（已有）：**
```sql
CREATE TABLE IF NOT EXISTS bookings (
  id TEXT PRIMARY KEY,
  userId TEXT NOT NULL,
  courtyardId TEXT NOT NULL,
  courtyardName TEXT,
  userName TEXT,
  userPhone TEXT,
  date TEXT NOT NULL,
  timeSlot TEXT,
  slot TEXT,
  visitors INTEGER DEFAULT 1,
  code TEXT,
  status TEXT DEFAULT 'pending',
  checkedAt TEXT,
  createdAt TEXT NOT NULL DEFAULT (datetime('now')),
  updatedAt TEXT NOT NULL DEFAULT (datetime('now'))
);
```

### 6.2 需求备注

- **容量约束**：容量上限写死在 `booking-store.ts` 的 20 人/时段。Server 端 `bookings.js` 的 `POST /check` 核销接口无容量校验。MVP 阶段靠前端约束，以后需后移。
- **核销码生成**：`genCode()` 函数在前端生成，格式 `YY + Date.now().slice(-6) + random(10-99)`。Server 端仅在 POST /bookings 中由 crudRoutes 接受前端传入。
- **状态枚举不一致**：前端 type 为 `"pending" | "checked" | "cancelled" | "expired"`，但 seed 数据中使用了 `"confirmed"` —— 存在歧义，需要统一。

## 七、技术要点

### 7.1 架构红线：Server 端驱动

> 所有关键业务逻辑应在 Server 端执行，包括容量校验、核销码唯一性校验、状态变更。

**当前架构差距**：
- 容量校验完全在前端（`booking-store.ts` 第 57-62 行）
- 核销码生成在前端（`booking-store.ts` 第 63 行 `genCode()`）
- Server 端 `bookings.js` 仅提供通用 CRUD 路由 + POST /check 核销接口
- 预约创建由 `crudRoutes` 直接接收前端 payload，缺乏服务端校验层

**MVP 阶段可以接受**前端驱动的轻量实现，但架构设计需为后移预留空间。

### 7.2 定时任务

| 任务 | 频率 | 说明 |
|------|------|------|
| 预约过期扫描 | 每 5 分钟 | 扫描所有 `status = pending` 且 `date < 今天` 的记录 → 变更为 `expired` |

### 7.3 并发一致性问题

当前容量校验完全在前端（内存 store），没有服务端一致性保障。在并发场景下，多个用户可能同时预约同一时段，造成超卖。

**MVP 阶段不影响**：因为试用场景下单用户有限且院落容量非实时竞争。后续可引入：

- 写入时原子校验：`UPDATE bookings SET ... WHERE NOT EXISTS (SELECT 1 FROM bookings WHERE courtyardId=? AND date=? AND slot=? AND status!='cancelled' AND visitors_sum >= 20)`
- 或行级锁 + 条件更新

### 7.4 核销码唯一性

核销码由前端 `Date.now()` 生成，不确定性低，但理论上存在碰撞可能。Server 端应增加唯一性校验（查询已存在的 code），碰撞时自动重新生成。

### 7.5 状态枚举统一

前端类型定义和种子数据中使用了不同的状态值：
- 前端类型：`"pending" | "checked" | "cancelled" | "expired"`
- 种子数据：`"confirmed"`（对应 pending）

**建议统一为**：`pending | checked | cancelled | expired`，seed 中的 `confirmed` 改为 `pending`。

## 八、验收标准

### 8.1 主流程验收

- [ ] 用户能完整走完「浏览院落 → 选择时段 → 预约成功 → 查看核销码」
- [ ] 同一院落同日同时段总预约人数超过 20 时，新预约提示已满并禁止提交
- [ ] 预约成功后核销码正确展示，格式为 `YY` + 数字
- [ ] 用户可在「我的预约」中查看自己的所有预约记录
- [ ] 用户可取消 pending 状态的预约，取消释放时段容量（前端表现）
- [ ] 工作人员在桌面端输入核销码可完成核销，状态变更为 checked

### 8.2 异常流程验收

- [ ] 无效核销码核销提示「核销码无效」
- [ ] 已核销预约再次核销提示「该预约已核销」
- [ ] 已取消预约核销提示「该预约已取消」
- [ ] 已取消预约不可再次取消（UI 上取消按钮隐藏）

### 8.3 桌面端管理验收

- [ ] 管理员可查看所有预约记录，支持按院落、日期、状态筛选
- [ ] 管理员可输入核销码进行核销
- [ ] 管理员可手动取消 pending 状态的预约

### 8.4 非功能验收

- [ ] 核销码不重复（不同的预约生成不同的 code）
- [ ] 预约数据与打卡数据关联畅通（同一院落同时有打卡记录和预约记录不冲突）
- [ ] VR 游览页面全屏展示正常，全屏切换功能正常
