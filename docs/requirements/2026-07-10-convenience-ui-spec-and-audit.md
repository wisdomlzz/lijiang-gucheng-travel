## 1. 总体发现

审计 40+ 个 UI 文件，发现 **~140 个问题**，按类别分布：

| 类别 | 数量 | 严重程度 |
|------|------|----------|
| 缺少加载态（Loading/Skeleton） | 25+ | 🔴 每页都有 |
| 缺少错误处理（Error Boundary / try/catch） | 20+ | 🔴 普遍 |
| 颜色硬编码，未用设计系统 Token | 50+ | 🟡 量大但对功能无影响 |
| 按钮缺少 loading/disabled 状态（防重复提交） | 15+ | 🟡 中等 |
| 空状态缺失 | 5 | 🟡 中等 |
| 交互假实现（点击无实际效果） | 5 | 🟡 中等 |
| 按钮条件逻辑错误 | 3 | 🔴 影响功能 |
| Demo 调试代码残留 | 2 | 🟢 低 |
| 缺少 Error Boundary | 全部 | 🔴 每页都有 |

**最严重的问题**: 所有页面都没有 Error Boundary，一旦某个组件崩溃，整个页面白屏。

## 2. 跨端通用问题

### 2.1 必须修复的架构问题

| # | 问题 | 影响范围 | 修复方案 |
|---|------|----------|----------|
| 1 | **所有页面无 Error Boundary** | 全部页面 | 创建 `ErrorBoundary` 组件，包裹每个页面路由 |
| 2 | **所有按钮无加载态** | 所有操作按钮 | 每个 async 操作按钮加 `disabled={loading}` + spinner |
| 3 | **所有页面无初始加载骨架屏** | 全部列表/详情页 | 列表用 `Skeleton` 组件，详情用 `CardSkeleton` |
| 4 | **颜色硬编码** | 全部页面 | 统一用 `bg-primary` / `text-primary` 等 Tailwind 类 |

### 2.2 B 端全局 branding 不一致

**问题**: 便民服务 B 端所有页面使用 `#F59E0B`（琥珀色）作为主色调，而设计系统 primary 是 `#2563EB`（丽江蓝）。

**建议**: 确认这是否是故意的 branding 差异。如果是，应定义 CSS 变量 `--convenience-primary: #F59E0B` 统一引用，而非每页硬编码。

### 2.3 状态标签颜色分散

**问题**: 状态标签颜色（`STATUS_MAP` / `STATUS_META` / `SERVICE_TYPE_COLORS`）在至少 6 个文件中重复定义，硬编码 hex 值。

**建议**: 抽取为 `src/features/convenience/shared/status-colors.ts` 统一导出。
