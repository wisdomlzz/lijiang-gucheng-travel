# 便民服务平台 — 需求澄清会议材料

> **会议日期**: 2026-07-10
> **项目定位**: 便民服务平台 MVP 1.0 生产级开发
> **当前状态**: 已有 Demo 版本，需求已定型，启动生产级重构

---

## 一、项目定位

**一句话**: 丽江古城内，游客/商户发起便民服务需求 → 系统派单给服务人员 → 上门服务 → 支付 → 评价，线上闭环。

**MVP 核心目标**:
1. 验证「下单→派单→接单→报价→支付→服务→完成→评价」主线闭环
2. 验证双支付方案（线上 + 现金）可行性
3. 攒第一批真实订单 + 评价数据

---

## 二、三类角色

| 角色 | 使用端 | 核心诉求 |

## 四、核心业务流程

### 4.1 主流程

```
用户下单（选服务类型+地址+备注，不选支付方式）
    ↓ createOrder
S10 已下单
    ↓ Server 自动派单（每 10 秒）
A20 已指派（staff 待接单）
    │
    ├─ 接单 → A30 已接单
    │     ↓ 到场打卡（arriveCheckin，不改状态，只写 arrivedAt）
    │     ↓ 师傅报价（前置条件：arrivedAt 不为空）
    │   A35 已核价
    │     ↓ 用户确认报价 + 选择支付方式（锁定 paymentMethod）
    │     │
    │     ├─ 线上支付 → 用户线上付钱 → A40
    │     └─ 现金支付 → 师傅确认现金已收 → A40
    │   A40 已收款
    │     ↓ 开始服务
    │   S48 服务中
    │     ↓ 完工上传照片
    │   S55 完工待确认
    │     ↓ 用户确认 / 24h 自动确认
    │   S40 已完成
    │     ↓ 用户评价
    │   流程结束
    │
    └─ 拒单 → 回 A10，重新派单（最多 3 次重试，失败转 S90）
```

### 4.2 11 个状态码一览

| 状态码 | 名称 | 阶段 | 说明 |
|--------|------|------|------|
| S10 | 已下单 | 派单 | 用户提交，等待系统派单 |
| A10 | 待重派 | 派单 | 拒单/失败，重回派单池 |
| A20 | 已指派 | 派单 | 已派给师傅，待接单 |
| A30 | 已接单 | 服务准备 | 师傅接单，等到场 |
| A35 | 已核价 | 支付 | 师傅报价完成，等用户确认+支付 |
| A40 | 已收款 | 支付 | 用户已付款（线上/现金） |
| S48 | 服务中 | 服务 | 师傅正在服务 |
| S55 | 完工待确认 | 验收 | 师傅标记完工，等用户确认 |
| S40 | 已完成 | 完成 | 终态 |
| S50 | 已取消 | 终止 | 终态 |
| S90 | 待人工 | 异常 | 异常，等待管理员处理 |

## 五、双支付方案（强需求，不能砍）

### 5.1 核心设计

- 下单时**不选**支付方式
- 师傅上门报价后，用户与师傅协商 → 锁定 **paymentMethod**
- 锁定后**不可修改**

### 5.2 五层防打架

| 层级 | 机制 |
|------|------|
| 第一层 | A35 且未锁定前，不开放任何支付/收款按钮 |
| 第二层 | 锁定 online：B 端无收款按钮；锁定 cash：C 端无线上支付按钮 |
| 第三层 | Server 校验 paymentMethod 与请求方匹配 |
| 第四层 | A40 后不能再支付/确认（幂等） |
| 第五层 | 所有支付操作记操作日志 + 支付流水 |

### 5.3 结算规则

- **线上支付**: T+7 后进入可提现余额，师傅申请提现 → 管理员审批 → 平台打款
- **现金支付**: 师傅直接收全款，不计入可提现余额，仅记录流水/业绩
- **MVP 无抽成**: 用户付多少，师傅拿多少

## 六、异常处理（4 条线）

### 6.1 取消线

| 状态 | 规则 |
|------|------|
| S10/A10 | 用户直接取消 → S50，**不扣费** |
| A20（已指派） | 因已指派 staff，需走申请取消流程（A30+ 规则），或由管理员 forceCancel |
| A30+ | 用户申请 cancelRequested → 管理员审批 |
| 管理员审批 | 同意 → S50（扣费按比例计算，可调），拒绝 → 清 cancelRequested，恢复服务 |
| S40 已完成 | 不走取消，走售后/投诉 |

### 6.2 派单异常线

- A20 超 5 分钟未接 → 自动回 A10 重派
- 累计 3 次重派失败 → S90
- 管理员可在 S90 手动重派 或 forceCancel

### 6.3 报价争议线

- 用户拒绝报价 → S90（记 beforeManualStatus = A35）
- 管理员协调成功 → 恢复 A35（重置 quotedAt）
- 协调失败 → forceCancel → S50（扣费 = 0，管理员可调）

### 6.4 投诉线

- 用户提交投诉 + 证据 → C10 待审核
- 管理员审核 → C40 成立（记投诉数、处罚分）或 CR 驳回
- 投诉成立不影响好评率

## 七、功能模块清单

### C 端（游客/商户）
1. 服务浏览与下单（选类型、填地址、备注）
2. 订单跟踪（看状态、师傅信息、联系师傅）
3. 确认报价 / 拒绝报价
4. 线上支付（模拟）
5. 确认完成 / 评价（评分+文字+图片）
6. 申请取消订单
7. 投诉入口

### B 端（服务人员）
1. 接单 + 拒单（选原因）
2. 到场打卡（记录 arrivedAt）
3. 报价（未打卡置灰提示）
4. 确认现金收款
5. 开始服务 / 完工上传照片
6. 提现申请
7. 入驻申请
8. 通知查看
9. 收入统计

### 桌面端（管理员）
1. 订单管理（查看、筛选、手动派单、强制取消）
2. S90 人工处理池（展示进入前状态+异常原因）
3. 取消审批（含扣费计算、可调）
4. 服务人员管理（入驻审核、增删改、分配区域、禁用+处理进行中订单弹窗）
5. 结算管理（收入明细、提现审批、财务报表）
6. 评价管理（查看、回复、删除违规）
7. 投诉处理（审核、成立/驳回）
8. 系统配置（取消扣费规则、超时时间、派单参数）
9. 片区/站点管理

### 系统能力
1. 自动派单（Server 定时任务，每 10 秒）
2. 接单超时检测（5 分钟未接 → 回 A10）
3. 支付超时检测（30 分钟未付 → S90）
4. 自动确认完工（24 小时未确认 → S40）
5. 好评率统计（定时计算）
6. 今日订单数清零（每天）
7. 结算 T+7
8. 订单操作日志（所有 transition 自动记）

## 九、技术架构红线

1. **所有业务逻辑、状态流转、定时任务必须在 Server 端**，前端只负责展示和发请求
2. Server 端定时任务跑满 7 个（自动派单/接单超时/支付超时/自动确认/好评率/清零/结算）
3. 所有状态变更记操作日志（order_operation_logs）
4. 支付接口幂等（重复请求不重复扣款）
5. 派单并发约束：每日接单上限用条件更新防止超量

## 十、现状与生产级开发的差距

| 领域 | Demo 现状 | 生产级要求 |
|------|-----------|-----------|
| 认证 | 手机号无密码验证 | JWT 完整鉴权 + 短信验证 |
| 真实支付 | 模拟支付标记 | 微信/支付宝对接 |
| Server 定时 | 单进程 setInterval | 分布式定时任务 + 幂等 |
| 派单算法 | Haversine + zone | 专业 Geo 索引（如 PostGIS） |
| 并发控制 | 无锁 | 行锁/乐观锁 |
| 监控告警 | 无 | 服务监控 + 错误告警 |
| 日志 | 基础 log | ELK/结构化日志 |
| 部署 | 单机 demo | 容器化 + CI/CD |
| 退款 | T+7 前简易处理 | 完整退款流程 + 财务冲正 |

> ***注**: 以上为从 Demo 到生产的普适性差距。具体项目按实际资源决定哪些优先级提升。*
