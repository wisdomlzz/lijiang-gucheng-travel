## 一、MVP 定位与边界

### 1\.1 我们在做什么

我们做的是 **便民服务平台 MVP 1\.0 版本**——不是完美的终态产品，而是第一个能跑通核心流程、能上线验证的最小版本。

**核心目标：**

- ✅ 验证「用户下单 → 师傅接单 → 上门服务 → 收钱 → 评价」这条主线能不能跑通

- ✅ 验证用户愿不愿意用、师傅愿不愿意接

- ✅ 攒第一批真实订单数据和评价数据

- ✅ 验证支付和结算模式的可行性

### 1\.2 MVP 原则：什么必须做，什么可以等

|优先级|原则|说明|
|---|---|---|
|🔴 必须|**主流程必须闭环**|下单→派单→服务→支付→评价，每一步都不能断|
|🔴 必须|**数据结构必须完整**|表结构要设计对，不然后期改数据成本极高|
|🔴 必须|**Server 端驱动状态机**|绝对不能靠前端定时器，这是架构红线|
|🟡 可以简化|异常流程可以人工兜底|比如派单失败、报价争议，先人工处理|
|🟡 可以简化|算法可以先做简单版|派单不用智能，按距离\+好评率排就行|
|🟢 以后做|体验优化类功能|比如消息推送、位置追踪、优惠券|
|🟢 以后做|复杂的体系化功能|比如抽成体系、诚信分、抢单模式、多级分销|

### 1\.3 明确不做的（MVP 边界）

以下功能 **1\.0 绝对不做**，避免范围蔓延：

- ❌ 平台抽成体系（MVP 阶段不抽成，先跑通流程）

- ❌ 保证金/押金制度（太重量级，后期再加）

- ❌ 智能派单算法（只做简单距离\+好评率匹配）

- ❌ 复杂诚信分体系（只记录好评率）

- ❌ 实时提现（周结/月结）

- ❌ 用户自选师傅

- ❌ 师傅抢单模式

- ❌ 到场 GPS 校验（纯打卡即可）

- ❌ 报价自动审核（用户不满意才转人工）

- ❌ 电子发票、保险对接

- ❌ 多级代理商体系

- ❌ 服务过程实时位置追踪

- ❌ 改单/加项线上流程（转人工客服）

## 二、核心用户角色

### 2\.1 三类角色

|角色|端|核心诉求|
|---|---|---|
|**C 端用户**|小程序/H5|快速找到靠谱师傅，价格透明，服务有保障|
|**B 端服务人员（staff）**|小程序/APP|接到更多订单，按时拿到钱|
|**平台管理员**|桌面端后台|管订单、管人员、管钱、处理异常|

### 2\.2 服务类型与服务人员模式

**六类便民服务（MVP 覆盖范围）：**

|服务类型|说明|典型场景|
|---|---|---|
|建筑垃圾清运|装修、拆改产生的建筑垃圾清运|家庭装修、商铺装修|
|生活垃圾清运|日常生活垃圾清运、大件垃圾处理|小区、写字楼垃圾清运|
|送水服务|桶装水、瓶装水配送上门|家庭、办公室送水|
|布草配送|酒店、民宿等布草（床单、毛巾等）收送|酒店、民宿、美容院|
|行李搬运|行李、包裹搬运服务（同城/跨城）|搬家、行李托运|
|送货服务|同城货物、包裹配送上门|电商落地配、同城跑腿|

**服务人员模式：**

所有服务人员默认走 **合作服务商模式**，按单结算。`staff_type` 字段保留，用于兼容未来可能的直营员工场景。

|staff\_type|说明|结算方式|
|---|---|---|
|partner|合作服务商（默认）|按单结算（MVP 阶段无抽成）|
|employee|直营员工（预留）|工资体系（暂不实现）|

- **身份**：外部合作的个体户、小团队或服务商

- **管理方式**：平台制定服务标准，服务人员接单执行

- **结算方式**：按单结算，MVP 阶段平台不抽成

- **接单权限**：可以选择接单或拒单，拒单过多影响派单优先级

## 三、核心业务流程

### 3\.1 下单主流程（主线必须闭环）

```
用户打开服务页
    ↓
选择服务类型 + 填写地址备注
    ↓ 提交订单
【S10 已下单】
    ↓ Server 定时任务触发自动派单
【A20 已指派】
    ↓ 师傅接单 / 拒单回 A10 重派
【A30 已接单】
    ↓ 师傅到场打卡（仅记录 arrived_at，无需GPS/照片/备注）
【仍为 A30，已记录到场时间】
    ↓ 师傅报价（Server 校验 arrived_at 不为空）
【A35 已核价】
    ↓ 用户与师傅确认最终价格和支付方式
    ├─ 用户接受报价 → 选择线上支付/现金支付，并锁定支付方式
    └─ 用户拒绝报价 → 转 S90 人工处理
    ↓
    ├─ 线上支付 → 用户线上付钱
    └─ 现金支付 → 用户付现金给师傅，师傅确认收款
    ↓
【A40 已收款】
    ↓ 开始服务
【S48 服务中】
    ↓ 完工上传照片
【S55 完工待确认】
    ↓ 用户确认完成 / 24h 自动确认
【S40 已完成】
    ↓ 用户评价
生成评价记录 → 更新好评率
    ↓
流程结束
```

### 3\.3 结算与评价流程

> 💡 **MVP 阶段无抽成**：用户付多少，师傅拿多少。平台先不赚钱，跑通流程为主。
> 
> 

```
订单完成（S40）
    ├─→ 订单进入待评价状态
    │      ↓
    │   用户提交评价后写入 reviews 表
    │      ↓
    │   更新 staff 好评率
    │
    └─→ 写入 income_records 表
           ↓
        按支付方式区分结算口径
           ├─ 线上支付：全额计入师傅待结算收入，T+7 后进入可提现余额
           │      ↓
           │   staff 申请提现 → 管理员审批 → 平台打款
           │
           └─ 现金支付：仅计入师傅业绩/GMV/流水，不进入可提现余额
                  ↓
               平台不打款，仅做对账记录
```

## 四、双支付方案设计（强需求）

> 💰 **重要前提：现金 \+ 线上支付都是客户强需求，不能砍，必须都做，而且不能打架。**
> 
> 

### 4\.1 为什么会打架？

现金支付和线上支付都要做，但不能让两条收款路径同时打开。

如果系统在报价后同时允许：

```
A35 → C端用户线上支付 → A40
A35 → B端staff标记现金已收 → A40
```

就会出现两个问题：

- 用户线上付了钱，staff 又点了「现金已收」→ 系统以为收了两笔

- 或者两边都以为对方会操作，结果都没操作 → 订单卡死

所以支付方式不能长期处于“未锁定但两边都能操作”的状态。

### 4\.2 解决方案：报价协商后锁定支付方式

**核心思路：下单时不选择支付方式；服务人员上门后报价，用户与服务人员确认最终价格和支付方式，确认后由系统锁定支付方式。**

```
用户下单
    ↓
不选择支付方式，只提交服务类型、地址、备注
    ↓
师傅接单并到场
    ↓
师傅报价，用户与师傅协商最终价格和支付方式
    ↓
用户确认报价和支付方式
    ├─ 线上支付
    │   ├─ C端显示支付按钮
    │   └─ B端不显示现金收款按钮
    │
    └─ 现金支付
        ├─ C端不显示线上支付按钮，只显示「请付现金给师傅」
        └─ B端显示「确认现金已收」按钮
```

> 支付方式一旦在报价确认阶段锁定，就不允许再修改。MVP 不提供管理员改支付方式能力。

### 4\.3 线上支付流程

1. 用户下单时不选择支付方式

2. 师傅到场后报价，用户与师傅确认最终价格

3. 用户选择「线上支付」并确认报价，系统锁定 `payment_method = online`

4. 用户端显示「确认报价并支付」

5. 用户点击 → 调起微信/支付宝

6. 支付成功 → 第三方回调 Server → 订单变 A40 → 记录支付流水

7. B 端师傅看不到现金收款按钮，只显示「用户已支付」

### 4\.4 现金支付流程

1. 用户下单时不选择支付方式

2. 师傅到场后报价，用户与师傅确认最终价格

3. 用户选择「现金支付」并确认报价，系统锁定 `payment_method = cash`

4. 用户端显示「请向服务人员支付现金 XX 元」

5. C 端不显示线上支付按钮，不能再切换线上支付

6. 师傅收到现金后，在 B 端点「确认现金已收」，输入实收金额

7. Server 更新订单为 A40，记录现金收款信息（谁收的、什么时候、多少钱）

8. 用户端同步显示「已支付（现金）」

> 💡 **MVP 阶段无抽成**：现金支付师傅直接收全款，平台不抽成，不存在抽成对账问题。
> 
> 

### 4\.5 防打架机制（五层防护）

|层级|机制|说明|
|---|---|---|
|第一层|阶段隔离|报价确认前不开放任何支付/收款按钮|
|第二层|UI 隔离|锁定线上支付后：B端无收款按钮；锁定现金支付后：C端无线上支付按钮|
|第三层|Server 校验|所有支付/收款请求校验 payment\_method，不匹配直接拒绝|
|第四层|状态幂等|A40 状态的订单不能再次支付或确认收款，重复请求直接返回成功|
|第五层|操作日志|所有支付操作都记日志：谁、什么时候、做了什么、多少钱|

### 4\.6 财务对账

后台必须基于 `payment_records` 生成三类流水/报表，**线上和现金严格分开统计**：

|报表|内容|
|---|---|
|线上支付流水表|订单号、支付方式、金额、时间、第三方交易号|
|现金收款记录表|订单号、收款人、金额、时间|
|交易流水/GMV统计表|按日/周/月统计：线上支付GMV、现金支付GMV、订单量、平台实际收入|

> 💡 MVP 阶段无抽成，平台实际收入为 0；这里统计的是交易流水和 GMV，不是平台收入。
> 
> 

### 4\.7 支付方式能改吗？

**不能改。**

支付方式在报价确认阶段由用户确认并锁定，锁定后不允许用户、师傅或管理员修改。

如用户现场临时想更换支付方式，MVP 不做线上改支付方式流程：

- 未收款前：可取消当前确认动作，重新确认报价和支付方式（仅限尚未锁定前）
- 已锁定后：不允许修改
- 已收款后：不允许修改，只能走退款/售后/重新下单等人工处理

> 砍掉“管理员改支付方式”，避免账务冲正、流水混乱和双重收款风险。

## 五、订单状态机设计

### 5\.1 状态列表

|状态码|状态名|说明|阶段|
|---|---|---|---|
|S10|已下单|用户提交订单，等派单|派单阶段|
|A10|待重新派单|拒单/派单失败，重回派单池|派单阶段|
|A20|已指派|系统派给师傅，等接单|派单阶段|
|A30|已接单|师傅确认接单，等到场|服务准备|
|A35|已核价|师傅报完价，等用户确认报价和支付方式|支付阶段|
|A40|已收款|用户已付钱（线上/现金）|支付阶段|
|S48|服务中|师傅正在服务|服务阶段|
|S55|完工待确认|师傅标记完工，等用户确认|验收阶段|
|S40|已完成|用户确认，订单结束|完成|
|S50|已取消|订单已取消|终止|
|S90|待人工处理|异常，等管理员处理|异常|

### 5\.2 状态流转规则

> 🔴 **架构红线：所有状态变更必须由 Server 端触发，前端只能发请求，不能直接改状态。**
> 
> 

|源状态|目标状态|触发动作|触发方|
|---|---|---|---|
|\-|S10|createOrder|C端用户|
|S10|A20|autoDispatchOrder|Server 定时任务|
|A20|A30|acceptOrder|B端师傅|
|A20|A10|rejectOrder / 超时未接|B端 / Server|
|A10|A20|autoDispatchOrder|Server 定时任务|
|A10|S90|重试3次失败|Server|
|A30|A35|submitQuote（需先完成到场打卡）|B端师傅|
|A35|A40|确认报价并支付 / 确认现金已收|C端 / B端|
|A35|S90|用户拒绝报价|C端用户|
|A35|S90|收款确认超时30分钟（线上未支付/现金未确认收款）|Server|
|A40|S48|startService|B端师傅|
|S48|S55|completeService \+ 传照片|B端师傅|
|S55|S40|confirmComplete / 24h自动确认|C端 / Server|
|S10/A10/A20|S50|用户直接取消|C端用户|
|A30\+|S50|用户申请 \+ 管理员审批|C端 \+ 管理员|
|未完成状态（除S40/S50外）|S50|forceCancel|管理员|
|S90|A10/A20|派单失败后重新派单|管理员|
|S90|A30/A35|报价争议协调后重新报价/继续确认|管理员|
|S90|A35|支付超时后允许继续支付|管理员|
|S90|进入S90前状态|取消申请驳回/异常解除|管理员|
|S90|S50|人工取消|管理员|


### 5.3 端侧展示与可操作性约束

#### 5.3.1 到场打卡与报价
- `submitQuote` 的前置条件为：`arrived_at` 不为空。服务人员端在未打卡前不得发起报价提交；报价入口需给出明确引导（例如置灰并提示“请先完成到场打卡”）。

#### 5.3.2 A35 收款确认超时（线上/现金统一口径）
- 订单处于 `A35 已核价` 时，若超过 `pay_timeout_minutes` 未完成收款确认（线上未支付或现金未确认收款），系统自动转入 `S90 待人工处理`，并写入 `manual_reason = pay_timeout`。

#### 5.3.3 用户拒绝报价（A35 → S90）后的师傅端表现
- 当用户拒绝报价导致订单进入 `S90 待人工处理` 后：
  - 该订单从师傅“进行中/待处理任务”列表中移除；
  - 师傅侧仅允许只读查看（如保留历史记录需要），不得继续报价、确认收款、完工等操作；
  - 后续仅由管理员/客服在 S90 人工处理池中执行“重新报价/继续确认/确认取消”等动作。

#### 5.3.4 报价被拒绝后的取消口径（与“取消申请”区分）
- 用户拒绝报价本身不等同于取消订单，订单先进入 `S90` 由客服协调。
- 若客服协调失败需要取消，管理员在 S90 池执行 `forceCancel`（订单进入 `S50 已取消`）。该类取消默认 `cancel_fee = 0`，管理员可在审批时手动调整。

## 六、功能模块清单

### 6\.1 P0 必须有（MVP 缺一不可）

#### C 端用户

- 服务浏览与下单（选类型、填地址；支付方式在上门报价确认后选择）

- 订单跟踪（看状态、看师傅信息、联系师傅）

## 八、技术架构要点

### 8\.1 架构红线：必须 Server\-Driven

> 🔴 **再说一遍：所有业务逻辑、状态流转、定时任务必须在 Server 端，前端只负责展示和发请求。**
> 
> 

**为什么不能用前端定时器：**

- 用户关页面/切后台，定时器就停了，订单直接卡死

- 多端不一致（A设备下单，B设备打开，定时器没跑）

- 不可控，没法统一监控和告警

### 8\.2 定时任务清单（全部 Server 端）

|任务|频率|说明|
|---|---|---|
|自动派单|每10秒|扫描 S10/A10 订单，执行派单|
|接单超时检测|每分钟|A20 超过5分钟未接，自动回 A10|
|支付超时检测|每分钟|A35 超过30分钟未完成收款确认（线上未支付/现金未确认收款），转 S90|
|自动确认完工|每小时|S55 超过24小时未确认，自动转 S40|
|好评率统计|每天凌晨|根据 reviews 重新计算所有师傅的好评率；投诉不直接改 good_rate|
|今日订单数清零|每天凌晨|重置 today\_orders 字段|

### 8\.3 派单算法（MVP 简化版）

**目标：按距离优先，同时尽量让人人有单做，不要某些人忙死某些人闲死。**

**算法逻辑：**

1. **筛选条件**：

    - 在线状态 = 在线

    - 服务类型匹配

    - 服务区域匹配

    - 今日接单未达上限（可配置）

2. **排序规则（综合得分）**：

    - 距离权重：70%（越近得分越高）

    - 好评率权重：20%（好评率越高得分越高）

    - 负载权重：10%（今日接单数越少得分越高）

    - 综合得分 = 距离得分*0\.7 \+ 好评率得分*0\.2 \+ 负载得分\*0\.1

3. **派单逻辑**：

    - 取综合得分最高的 1 个派单

    - 拒单就派给下一个

    - 最多重试 3 次，失败转 S90 人工处理

> 💡 **简单有效**：不用搞复杂的负载均衡，加个今日接单权重就行，既能保证距离优先，又能避免单都被某几个师傅抢走。
> 
> 

### 8\.4 派单规则

|维度|规则|
|---|---|
|派单方式|自动派单，服务人员可以选择接单或拒单|
|接单时限|5分钟内未接单自动重派|
|拒单处理|拒单后自动派给下一个，累计3次失败转人工|
|优先级|综合得分排序：距离70% \+ 好评率20% \+ 负载10%|
|拒单惩罚|基于 dispatch_logs 统计拒单率/超时率，过高则降低派单优先级（可配置阈值）|
|每日上限|可设置每人每日最大接单量（防止过劳/不公平）|

### 8\.5 并发一致性要求（每日接单上限强约束）

- **强约束定义**：任何并发情况下，单个师傅当日成功接单数不得超过配置上限。
- **生效点**：以 `acceptOrder` 成功为准进行原子校验与计数（不以派单筛选结果作为最终约束）。
- **实现要求（二选一）**：
  - 行锁：对 `staffs` 行加锁（`SELECT ... FOR UPDATE`）后再判断并更新；
  - 条件更新/乐观锁：使用 `today_orders < limit` 作为更新条件，更新成功才算接单成功。

## 九、结算设计

> 💡 **MVP 阶段无平台抽成**：用户付多少，师傅拿多少。先跑通流程，验证业务，后期再加抽成。
> 
> 

### 9\.1 资金流向

```
用户支付（线上/现金）
      ↓
线上支付 → 资金进入平台账户
现金支付 → 师傅直接收全款
      ↓
订单完成后，均计入师傅业绩收入和平台 GMV
      ↓
按支付方式区分结算：
      ├─ 线上支付：T+7 天后进入可提现余额
      │      ↓
      │   服务人员申请提现 → 管理员审批 → 平台打款
      │
      └─ 现金支付：不进入可提现余额
             ↓
          平台不打款，仅记录流水和业绩
```

### 9\.2 线上支付结算

1. 用户线上支付成功 → 资金进入平台账户

2. 订单完成（S40 状态）→ 全额计入师傅待结算收入

3. T\+7 天后（过了投诉期）→ 进入可提现余额

4. 师傅申请提现 → 管理员审批 → 打款

### 9\.3 现金支付结算

1. 师傅直接收取现金（全额）

2. 订单完成后，系统记录收入流水（仅对账用）

3. 因为 MVP 无抽成，所以现金支付不存在平台抽成对账问题

4. 现金支付订单只计入师傅业绩收入、平台 GMV 和交易流水，不进入师傅可提现余额

5. 师傅收多少都是自己的，平台不参与现金资金流转，也不对现金订单发起打款

> 💡 **为什么现金不用对账？** 因为 MVP 阶段不抽成，平台不碰现金的钱，师傅收多少都是自己的。平台只记录流水，方便统计和客服查单。
> 
> 

### 9\.4 结算周期

- **可提现时间**：订单完成后 T\+7 天进入可提现状态（预留投诉处理期）

- **提现周期**：周结（每周可申请一次）或月结（每月固定日期）

- **提现门槛**：可设置最低提现金额（如 100 元），后台可配置

- **到账时间**：审批通过后 1\-3 个工作日到账

### 9\.5 结算报表

后台需提供以下结算报表：

|报表|内容|
|---|---|
|服务人员账户明细|每笔收入、每笔提现、余额变动|
|平台流水统计|按日/周/月：订单总额、线上收入、现金收入、订单量|
|提现审批列表|待审批、已通过、已驳回、已打款|

## 十、服务人员入驻流程

### 10\.1 入驻流程

```
师傅下载/打开B端
    ↓
填写注册信息（手机号、姓名、身份证）
    ↓
上传证件照片（身份证正反面）
    ↓
选择服务类型 + 服务区域
    ↓
提交入驻申请
    ↓
【待审核】
    ↓
管理员后台审核
    ├─ 审核通过 → 账号开通 → 可以接单
    └─ 审核驳回 → 填写驳回原因 → 师傅修改后重新提交
```

### 10\.2 入驻审核要点

管理员审核时需要确认：

- ✅ 身份证信息真实有效

- ✅ 证件照片清晰可辨

- ✅ 服务类型和区域合理

- ✅ 手机号能正常联系

> 💡 **MVP 阶段**：人工审核就行，不用搞什么人脸识别、公安接口对接，太重了。
> 
> 

### 10\.3 账号状态

|状态|说明|能否接单|
|---|---|---|
|待审核|提交申请，等审核|❌|
|已驳回|审核不通过，需修改|❌|
|正常|审核通过，正常使用|✅|
|已禁用|管理员禁用账号|❌|

|S48 服务中|❌ 基本不能|特殊情况找客服|协商|
|S55 待确认|⚠️ 申请|用户申请 \+ 管理员审批|协商|
|S40 已完成|❌ 不能|走售后/投诉/退款/财务冲正，不回退订单主状态|\-|
|S50 已取消|\-|\-|\-|
|S90 待人工|✅ 可以|管理员操作|看情况|

### 11\.2 扣费规则（后台可配置）

> 所有扣费比例都在后台配置，不用写死在代码里。
> 
> 

|配置项|说明|默认值|
|---|---|---|
|cancel\_fee\_before\_accept|接单前取消扣费比例|0%|
|cancel\_fee\_after\_accept|接单后未到场取消扣费比例|10%|
|cancel\_fee\_after\_pay|支付后取消扣费比例|20%|
|cancel\_fee\_min\_amount|最低取消费用|5元|
|cancel\_fee\_max\_amount|最高取消费用|50元|

**扣费计算逻辑：**

1. 根据订单状态找到对应扣费比例

2. 扣费金额 = 订单金额 × 扣费比例

3. 如果低于最低取消费，按最低收取

4. 如果高于最高取消费，按最高收取

5. 管理员审批时可以手动调整扣费金额

### 11\.3 退款处理

#### 11\.3\.1 基本原则
- 退款属于资金动作，订单主状态按业务事实保持一致：
  - **取消订单**：订单主状态为 `S50 已取消`；
  - **已完成订单的售后/投诉退款**：订单主状态保持 `S40 已完成`（不回退）。
- 退款成功后必须写入资金流水（线上）或退款记录（现金），并记录操作日志，保证可追溯。

#### 11\.3\.2 线上支付退款
- **退款方式**：原路退回（微信/支付宝）。
- **退款时效**：1-3 个工作日到账。
- **是否支持部分退款**：不支持（仅支持按规则计算后的单次退款）。
- **流水记录（payment_records）**：
  - 退款时新增一条退款流水：`amount` 为负数，`origin_payment_id` 指向被退款的原支付流水；
  - 原支付流水的 `status` 更新为 `refunded`。

#### 11\.3\.3 现金支付退款
- **退款方式**：师傅线下直接退现金给用户。
- **平台记录**：平台侧需记录一条“现金退款记录”（可复用 `payment_records`，写入负数金额、`payment_method=cash`，并记录退款确认人/时间），用于客服查单与统计。

#### 11\.3\.4 已进入可提现余额后的退款（财务冲正）
- 该场景不做自动扣回/冻结等复杂资金逻辑。
- 发生时订单进入 `S90 待人工处理`，由财务/管理员人工处理冲正，并在后台备注与日志中完整记录处理结果。

## 十二、验收标准

### 12\.1 主流程验收

* [ ] 用户能完整走完「下单→派单→接单→报价→支付→服务→完工→确认→评价」

* [ ] 用户可以拒绝报价，拒绝后订单转 S90 人工处理

* [ ] 线上支付订单：支付成功后状态正确，财务有流水

* [ ] 现金支付订单：师傅确认收款后状态正确，现金有记录

* [ ] 订单完成后进入待评价；用户提交评价后生成 review 记录，好评率正确更新

* [ ] 所有状态变更 Server 端驱动，刷新页面不丢失

* [ ] 服务人员能正常接单、结算，金额计算正确

### 12\.2 异常流程验收

* [ ] 师傅拒单后，订单自动重新派单

* [ ] 派单 3 次失败，转 S90 待人工

* [ ] S10/A20 状态用户可以直接取消

* [ ] A30\+ 状态取消需要管理员审批

* [ ] 管理员可以强制取消未完成订单；S40 已完成订单不回退主状态，只走售后/退款/投诉处理

* [ ] 支付超时转 S90，不自动取消

* [ ] 用户拒绝报价转 S90，管理员可以处理

* [ ] 取消扣费计算正确，后台可以配置规则

### 12\.3 后台管理验收

* [ ] 管理员可以查看所有订单，按状态筛选

* [ ] 管理员可以手动派单给指定师傅

* [ ] 管理员可以审批取消申请，调整扣费金额

* [ ] 管理员可以查看和回复评价

* [ ] 管理员可以查看收入明细，审批提现

* [ ] 财务报表区分线上收入和现金收入

* [ ] 服务人员入驻审核流程完整

* [ ] S90 人工处理池可以看到所有异常订单，并展示进入前状态、异常原因和可执行处理动作

* [ ] 系统配置可以在后台修改

### 12\.4 非功能验收

* [ ] 关了前端页面，自动派单、超时检测等定时任务正常跑

* [ ] 订单状态变更都有操作日志，可追溯

* [ ] 支付接口幂等，重复支付不会重复扣钱

* [ ] 一个订单出问题，不会影响整个系统

* [ ] 派单算法能兼顾距离和公平性，不会单都集中在少数人手里
