# 商城派单功能独立需求文档

> 口径说明：本需求不纳入当前新版 demo。当前 demo 只保留“商城管理后台”入口和“供应商入驻审核”。商品、订单、核销、物流、退款由后续商城后台承接；商城派单作为独立需求单独设计、开发、验收。

## 一、背景

平台后续将与商城系统对接。供应商在平台侧完成入驻审核后，通过单点登录进入商城后台，管理自己的商品、订单、核销、售后等业务。

当商城订单涉及线下服务履约时，需要独立的商城派单能力，将指定商品/SKU 的核销或预约订单分配给对应服务人员执行。该能力与本期“便民服务管理”的派单体系不同，不能混用。

## 二、范围边界

| 范围 | 是否属于本需求 | 说明 |
|---|---|---|
| 供应商入驻申请与审核 | 否 | 属于当前新版 demo 本期需求 |
| 商城后台单点登录入口 | 否 | 属于平台与商城对接基础能力 |
| 商品发布/商品管理 | 否 | 由商城后台承接 |
| 商品订单/支付/退款/物流 | 否 | 由商城后台承接 |
| 商城核销订单派单 | 是 | 本文独立需求 |
| 商城服务人员绑定 | 是 | 仅服务于商城派单 |
| SKU/时段派单规则 | 是 | 仅服务于商城派单 |
| 便民服务派单 | 否 | 属于当前 demo 的便民服务管理，不与商城派单混用 |

## 三、角色

| 角色 | 能力 |
|---|---|
| 平台管理员 | 配置商城派单规则、查看派单异常、人工改派 |
| 供应商管理员 | 查看本供应商商城服务订单、维护可服务人员、处理异常 |
| 商城服务人员 | 接收商城服务任务、完成核销后的服务履约 |
| 游客/买家 | 在商城下单、到场核销或预约服务 |

## 四、核心流程

1. 供应商在平台侧完成入驻审核。
2. 平台与商城打通账号和供应商身份映射。
3. 供应商通过 SSO 进入商城后台，维护商品、SKU、库存、订单。
4. 对需要线下履约的商品/SKU，配置是否启用派单。
5. 买家下单并支付。
6. 买家到店核销或预约到达服务时间。
7. 系统按 SKU、服务时段、服务人员绑定关系自动派单。
8. 派单失败进入人工处理。
9. 服务人员完成任务并回传服务结果。
10. 商城订单状态与平台侧服务履约状态同步。

## 五、功能清单

| 模块 | 功能 |
|---|---|
| 派单规则 | 按商品/SKU 配置是否派单、可服务时段、每时段接单上限 |
| 服务人员绑定 | 将服务人员绑定到商品/SKU，可设置优先级 |
| 时段排他 | 同一服务人员同一时段不可重复接相冲突任务 |
| 自动派单 | 核销或预约触发派单，按规则匹配服务人员 |
| 人工派单 | 自动失败后可人工指定、改派、取消 |
| 异常处理 | 无可用人员、时段冲突、超时未接、服务失败等 |
| 任务履约 | 服务人员接单、开始服务、完成服务、上传凭证 |
| 数据同步 | 与商城订单状态、核销状态、退款状态同步 |
| 权限控制 | 平台看全量，供应商只看本供应商数据 |

## 六、关键状态建议

| 状态 | 含义 |
|---|---|
| 待核销 | 商城订单已支付，等待买家到场核销 |
| 已核销待派单 | 核销完成，需要服务人员执行 |
| 待接单 | 已派给服务人员，等待接单 |
| 服务中 | 服务人员已接单并开始服务 |
| 已完成 | 服务履约完成 |
| 派单失败 | 无可用人员或规则冲突 |
| 已取消 | 订单取消或人工取消派单 |

## 七、与本期 demo 的关系

当前新版 demo 已删除商城核销派单、商城服务人员、SKU 派单配置相关页面和 mock 数据。本期保留：

| 本期保留 | 说明 |
|---|---|
| 供应商入驻 | C端提交，后台审核 |
| 商城管理后台入口 | 后续通过 SSO 打开商城后台 |
| 便民服务派单 | 独立于商城派单，仍属于本期 demo |
| 商家信息维护 | 购在古城、附近、酒吧/餐饮/住宿等信息维护保留 |
