## 一、定位与边界

### 1.1 我们在做什么

诚信分是对丽江古城平台上所有服务人员（便民师傅、讲解员、包车司机、旅拍等）的信用评级系统。每个服务人员有一个 0-100 分的诚信分值，基于用户评价、服务质量、投诉记录等动态调整。低于失信阈值的服务人员自动进入失信观察名单，限制其经营或接单权限。

**核心目标：**

- 为 C 端游客提供选择服务人员时的信用参考
- 为 B 端服务人员建立正向激励体系（分数高者获得更多派单机会）
- 为平台管理员提供失信管理工具，保障服务质量底线

### 1.2 诚信分 vs 好评率——本质区别

| 维度 | 诚信分 | 好评率 |
|------|--------|--------|
| 范围 | 0-100 分，可升可降 | 0%-100%，仅反映好评比例 |
| 粒度 | 多维度综合（评价、投诉、拒单、管理调整） | 单一维度（好评数/总评价数） |
| 动态性 | 每次事件（评价/投诉/拒单）实时增减 | 定期统计（每天凌晨重算） |
| 惩罚机制 | 低于阈值进入失信名单，限制接单/经营 | 无直接惩罚，仅影响派单权重 |
| 操作者 | 系统自动计算 + 管理员可手动调整 | 系统自动统计，管理员不可直接修改 |
| 定位 | 信用评级和准入管控 | 服务质量量化参考指标 |

**简单理解：** 好评率是"大家觉得他好不好"，诚信分是"平台觉得他靠不靠谱"。诚信分包含好评率不包含的维度（投诉、拒单、管理员处罚），且具有更强的管理约束力（失信名单限制经营）。

### 1.3 MVP 原则：什么必须做，什么可以等

| 优先级 | 原则 | 说明 |
|--------|------|------|
| 红色 | 数据模型必须完整 | trust_scores / score_rules / trust_thresholds 三张表结构要设计对 |
| 红色 | 后台管理闭环 | 管理员能查看所有服务人员评分、配置规则和阈值、手动调整分数 |
| 红色 | 诚信分必须 C/B 端可见 | C 端游客下单时能看到师傅诚信分；B 端师傅能看到自己的评分和状态 |
| 黄色 | 自动计算可以先做简化版 | 基于评价 + 投诉的自动增减，缺省规则写死，后台可配置 |
| 黄色 | 失信名单先做标记不做强制限制 | 标记失信状态，先展示给管理员，接单限制等强约束后续再做 |
| 绿色 | 智能调整算法以后做 | 自动恢复、观察期到期自动解除、申诉自动复审等 |
| 绿色 | 与派单深度绑定以后做 | 诚信分作为派单权重因子，影响派单优先级 |

### 1.4 MVP 明确不做

- 不做 C 端独立诚信分页面（仅嵌入在订单详情页或服务人员信息页）
- 不做 B 端独立诚信分页面（仅嵌入在个人中心 ServiceProfile 页面）
- 不做失信名单自动限权（限制接单自动拦截，先只做标记展示）
- 不做申诉/复审线上流程（管理员手动处理）
- 不做自动恢复调度任务（Server 定时任务，暂不实现）
- 不做诚信分历史趋势图/可视化报表
- 不做与派单算法的深度绑定

## 二、用户角色

### 2.1 涉及角色

| 角色 | 端 | 核心诉求 |
|------|----|----------|
| **C 端用户** | C 端小程序 | 下单前了解服务人员信用水平，选择靠谱的人 |
| **B 端服务人员（staff）** | B 端小程序 | 查看自己的诚信分、了解扣分原因、关注失信状态 |
| **平台管理员** | 桌面端后台 | 管理评分规则、查看全体人员评分、手动调整分数、处理失信人员 |

诚信分覆盖所有注册在平台上的服务人员，按角色标签分类：

| 角色标签 | 说明 |
|----------|------|
| 便民服务人员 | 便民送水/清运/布草/行李搬运/送货等 |
| 讲解员 | 古城讲解/导览服务 |
| 包车司机 | 古城周边/丽江旅游包车服务 |
| 旅拍 | 古城旅拍/摄影服务 |

## 三、核心业务流程

### 3.1 诚信分计算总流程

```
服务人员注册 → 获得初始分值（默认 100 分）
    ↓
每次用户评价
    ├─ 5 星评价 → +1 分
    ├─ 4 星评价 → +0.5 分
    └─ 1-3 星评价 → 不加减分
    ↓
每次投诉成立
    ├─ 一般投诉 → -3 分
    ├─ 严重投诉 → -6 分
    └─ 特别严重投诉 → -10 分
    ↓
每次拒单（超过阈值后）
    └─ 拒单率超过阈值 → 每次拒单 -2 分
    ↓
管理员手动调整
    └─ 调整范围：-5 到 +5 分/次，需填写原因
    ↓
分数更新 → 检查失信阈值
    ├─ 分数 >= 失信阈值 → 状态保持"正常"
    └─ 分数 < 失信阈值 → 状态变为"观察期"
        ↓
        ├─ 自动恢复开启 → 分数回升到恢复值以上时自动恢复
        └─ 自动恢复关闭 → 需管理员手动恢复
```

### 3.2 评价触发诚信分变更

```
C 端用户提交评价（1-5 星）
    ↓
Server 更新 order.review_status = done
    ↓
Server 写入 reviews 表
    ↓
Server 计算诚信分变更
    ├─ 5 星 → trustScore += 1
    ├─ 4 星 → trustScore += 0.5
    └─ 1-3 星 → 不变
    ↓
更新 trust_scores 表
    ├─ trustScore 字段
    ├─ rating5Count / rating4Count 等计数
    └─ scoreHistory 追加记录
    ↓
检查失信阈值 → 更新 status
    ↓
返回更新后的诚信分（用于前端实时展示）
```

### 3.3 投诉触发诚信分扣减

```
投诉被管理员审核通过（status → valid）
    ↓
根据投诉严重程度确定扣分
    ├─ 一般 → -3 分
    ├─ 严重 → -6 分
    └─ 特别严重 → -10 分
    ↓
更新 trust_scores 表
    ├─ trustScore -= 扣分值
    ├─ complaintCount += 1
    └─ scoreHistory 追加记录
    ↓
检查失信阈值 → 更新 status
    ↓
同步更新 staffs.complaint_count 和 penalty_score
```

### 3.4 管理员手动调整流程

```
管理员在后台选择服务人员
    ↓
输入调整分值（-5 到 +5）
    ↓
填写调整原因
    ↓
提交后
    ├─ 更新 trust_scores.trustScore
    ├─ 记录操作到 adjustRecords（谁调的、什么时候、调了多少、原因）
    ├─ scoreHistory 追加记录
    └─ 检查失信阈值 → 更新 status
```

### 3.5 失信观察期管理

```
服务人员诚信分低于失信阈值（默认 60 分）
    ↓
系统自动标记 status = "观察期"
    ↓
记录 observationStartAt
    ↓
后续行为
    ├─ 自动恢复模式（autoRecover = true）
    │   ├─ 分数回升到 recoverScore（默认 70 分）以上时自动恢复
    │   └─ status 恢复为"正常"
    │
    └─ 手动恢复模式（autoRecover = false）
        ├─ 分数回升后仍保持"观察期"状态
        └─ 需管理员手动操作恢复为"正常"
```

## 四、功能清单

### 4.1 P0 必须有（MVP 缺一不可）

#### 管理员后台

- 诚信评分规则 CRUD（扣分规则 + 加分规则的管理）
  - 规则类型：扣分（deduct）/ 加分（reward）
  - 规则字段：名称、触发条件、分值变化、启用/停用、说明
  - 操作：新增、编辑、删除、启用/停用、重置
- 失信阈值配置
  - 初始分值（默认 100）
  - 失信分数线（默认 60）
  - 自动恢复开关（默认开启）
  - 自动恢复分值（默认 70）
- 服务人员评分总览表
  - 表格字段：姓名、当前评分、状态（正常/观察期）、总订单、好评数、差评数、投诉次数、拒单次数
  - 根据阈值高亮显示正常/失信状态
- 管理员手动调整单个服务人员诚信分
  - 调整范围：-5 到 +5 分/次
  - 必填原因
  - 记录操作日志

#### 系统能力

- Server 端诚信分计算接口（评价提交后自动计算）
- Server 端投诉成立后自动扣分
- 操作日志记录（评分变更、管理员调整均需记录）

### 4.2 P1 建议有

- 服务人员评分历史记录查看（scoreHistory 时间线展示）
- C 端 TrustScoreBadge 组件（订单详情页展示服务人员诚信分徽章）
- B 端 ServiceProfile 诚信分进度条展示
- 管理员调整日志记录（score_adjust_records）
- 批量调整功能（管理员一次选择多人调整分数）

### 4.3 P2 以后做

- 失信名单自动限权（诚信分低于阈值时自动禁止接单）
- 观察期到期自动解除（定时任务扫描）
- 有利评价自动加分（已完成订单的评价自动触发加分）
- 申诉/复审线上流程
- 诚信分趋势图/可视化报表
- 诚信分与派单算法深度绑定（作为派单权重因子）
- 拒单自动扣分（拒单超过阈值后自动触发）
- 短信/消息通知（进入失信名单时通知服务人员）

## 五、数据模型

### 5.1 诚信分表（trust_scores）

| 字段 | 类型 | 说明 |
|------|------|------|
| staffId | TEXT PK | 服务人员 ID，主键 |
| supplierId | TEXT | 关联供应商 ID |
| name | TEXT | 服务人员姓名 |
| roleTag | TEXT | 角色标签（便民服务人员/讲解员/包车司机/旅拍） |
| trustScore | REAL | 当前诚信分（0-100），默认 100 |
| status | TEXT | 状态（正常/观察期） |
| totalOrders | INTEGER | 总订单数 |
| totalRatings | INTEGER | 总评价数 |
| rating5Count | INTEGER | 5 星评价次数 |
| rating4Count | INTEGER | 4 星评价次数 |
| rating3Count | INTEGER | 3 星评价次数 |
| rating2Count | INTEGER | 2 星评价次数 |
| rating1Count | INTEGER | 1 星评价次数 |
| complaintCount | INTEGER | 投诉成立次数 |
| rejectionCount | INTEGER | 拒单次数 |
| observationStartAt | TEXT | 进入观察期时间 |
| lastComplaintAt | TEXT | 最近投诉时间 |
| scoreHistory | TEXT (JSON) | 评分变更历史记录数组 |
| createdAt | TEXT | 创建时间 |
| updatedAt | TEXT | 更新时间 |

### 5.2 评分规则表（score_rules）

| 字段 | 类型 | 说明 |
|------|------|------|
| id | TEXT PK | 规则 ID |
| type | TEXT | 规则类型（deduct 扣分 / reward 加分） |
| name | TEXT | 规则名称 |
| condition | TEXT | 触发条件描述 |
| scoreChange | REAL | 分值变化（扣分为负数，加分为正数） |
| enabled | INTEGER | 是否启用（1 启用 / 0 停用） |
| description | TEXT | 规则说明 |
| createdAt | TEXT | 创建时间 |
| updatedAt | TEXT | 更新时间 |

### 5.3 失信阈值表（trust_thresholds）

| 字段 | 类型 | 说明 |
|------|------|------|
| id | INTEGER PK | 固定为 1，单行配置 |
| defaultScore | REAL | 初始分值，默认 100 |
| delinquentThreshold | REAL | 失信分数线，默认 60 |
| autoRecover | INTEGER | 是否自动恢复（1 自动 / 0 手动），默认 1 |
| recoverScore | REAL | 自动恢复分值，默认 70 |
| updatedAt | TEXT | 更新时间 |

### 5.4 诚信分调整记录表（score_adjust_records）

> 该表为 MVP 预留，当前 MVP 阶段调整记录仅在内存/前端 store 中维护，暂不落库。

| 字段 | 类型 | 说明 |
|------|------|------|
| id | TEXT PK | 记录 ID |
| staffId | TEXT | 被调整人 |
| operatorId | TEXT | 操作人 ID |
| operatorName | TEXT | 操作人姓名 |
| beforeScore | REAL | 调整前分数 |
| afterScore | REAL | 调整后分数 |
| reason | TEXT | 调整原因 |
| createdAt | TEXT | 调整时间 |

### 5.5 诚信分配置常量（硬编码，储存在前端）

| 配置项 | 默认值 | 说明 |
|--------|--------|------|
| maxScore | 100 | 诚信分上限 |
| minScore | 0 | 诚信分下限 |
| observationThreshold | 60 | 失信阈值（与数据库同步） |
| bonusRating5 | 1 | 5 星评价加分 |
| bonusRating4 | 0.5 | 4 星评价加分 |
| deductionComplaint.normal | -3 | 一般投诉扣分 |
| deductionComplaint.serious | -6 | 严重投诉扣分 |
| deductionComplaint.severe | -10 | 特别严重投诉扣分 |
| deductionRejection | -2 | 拒单扣分 |
| rejectionThreshold | 3 | 拒单扣分触发阈值（拒单超过此数才扣） |
| adminAdjustMin | -5 | 管理员调整下限 |
| adminAdjustMax | 5 | 管理员调整上限 |

## 七、前端 Store 设计

### 7.1 useTrustScoreStore（诚信分管理）

| 属性/方法 | 类型 | 说明 |
|-----------|------|------|
| scores | TrustScore[] | 所有服务人员诚信分数据 |
| adjustRecords | ScoreAdjustRecord[] | 管理调整记录（前端本地） |
| getScore(staffId) | TrustScore | undefined | 获取单个服务人员评分 |

## 九、当前实现 gap 分析

### 9.1 Server 端 gap

| 问题 | 描述 | 影响 | 建议修复 |
|------|------|------|----------|
| 无自动计算 | 评价提交后不会自动更新诚信分 | 诚信分始终为静态种子数据，不随实际业务变化 | 在评价提交 API 中增加诚信分计算逻辑 |
| 投诉扣分未联动 | 投诉成立后不会自动扣减诚信分 | 投诉与诚信分成两个独立系统 | 投诉审核通过时触发 trust_scores 更新 |
| 无自动恢复任务 | 没有定时任务扫描观察期人员 | 失信人员状态永远不变 | 增加 Server 定时任务（每天凌晨扫描） |
| adjustRecords 未落库 | 管理员调整记录仅在前端内存中 | 刷新页面后调整记录丢失，不可追溯 | 新增 score_adjust_records 表并落库 |

### 9.2 前端 gap

| 问题 | 描述 | 影响 | 建议修复 |
|------|------|------|----------|
| 无管理员调整 UI | 当前 TrustScoreConfigPage 表格中每行无"调整"按钮 | 管理员无法手动调整诚信分 | 表格每行增加调整按钮和弹窗 |
| 无 C 端 Badge | 无 TrustScoreBadge 组件嵌入 OrderDetailPage | 游客看不到服务人员诚信分 | 新增组件并嵌入 |
| 无 B 端进度条 | 无 ServiceProfile 诚信分嵌入 | 服务人员看不到自己诚信分 | 在个人中心页面嵌入 |
| 规则重置逻辑 | resetRules 仅清空前端列表，不恢复默认种子数据 | 重置后页面为空，需手动新增 | 重置时从 API 重新加载默认规则 |

### 9.3 数据一致性 gap

| 问题 | 描述 | 影响 | 建议修复 |
|------|------|------|----------|
| TRUST_SCORE_CONFIG 不匹配 | 前端硬编码的默认阈值（60）与数据库 seed 一致，但无自动同步机制 | 管理员修改阈值后，前端常量仍用旧值 | 阈值从数据库加载，不使用硬编码常量 |
| supplierId 关联 | trust_scores 的 supplierId 与 suppliers 表关联，但无外键约束 | 删除供应商可能导致孤儿数据 | 保持当前逻辑，MVP 不强制外键 |
| staff 与 trust_scores 对齐 | staff 表有 s1-s6，trust_scores 只有 s1-s4 | s5 杨送水、s6 周布草无诚信分记录 | 同步 seed 数据，确保所有 staff 都有初始诚信分 |

## 十、验收标准

### 10.1 功能验收

- [ ] 管理员可查看所有服务人员的诚信分列表（姓名、评分、状态、订单数、评价数、投诉数）
- [ ] 管理员可新增/编辑/删除/启用/停用评分规则
- [ ] 管理员可配置失信阈值（初始分值、失信分数线、自动恢复分值、自动恢复开关）
- [ ] 管理员可手动调整单个服务人员诚信分（-5 到 +5），调整原因必填
- [ ] 评分低于失信阈值的服务人员自动标记为"观察期"状态
- [ ] 自动恢复模式下，分数回升到恢复值以上时自动恢复为"正常"
- [ ] 阈值配置修改后，服务人员列表状态自动重新计算
- [ ] 诚信分变更记录在 scoreHistory 中可追溯

### 10.2 数据验收

- [ ] 种子数据包含 4 个不同状态的服务人员诚信分（含 1 个观察期人员）
- [ ] 种子数据包含 5 条评分规则（3 条扣分 + 2 条加分）
- [ ] 种子数据包含单行失信阈值配置
- [ ] Server 重启后种子数据不丢失（数据库持久化）
- [ ] 前端启动后从 API 加载所有诚信分和规则数据（hydration 正常）

### 10.3 非功能验收

- [ ] 所有诚信分数据变更都有操作记录（scoreHistory）
- [ ] 诚信分不得低于 0 分、不得高于 100 分
- [ ] 管理员调整范围限制在 -5 到 +5 分/次
- [ ] 前端页面响应式，在桌面端正常展示
- [ ] 诚信分数据与 Server 端同步，刷新页面不丢失
